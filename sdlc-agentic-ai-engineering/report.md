## 2. 상세 내용

### 2.1 문제 정의: 기존 AI 보조 개발의 한계

GitHub Copilot 같은 도구는 빠른 코딩을 가능하게 하지만 두 가지 핵심 문제를 만든다.

1. **이해도 격차**: 개발자가 AI가 생성한 코드를 충분히 이해하지 못한 채로 넘어가게 된다.
2. **PR 리뷰 부담**: 품질 안전장치 없이 빠르게 진행하면 리뷰어가 감당해야 할 기술 부채가 쌓인다.

"바이브 코딩" 또는 단일 에이전트 개발은 속도 면에서 매력적이지만, 제대로 된 견제와 균형(checks and balances) 없이는 "빠르게 움직이고 무언가를 부수는" 결과를 낳는다.

### 2.2 SDLC 프레임워크 설계

저자가 약 200시간의 Claude Code 사용 경험(2026년 2월 19일 시작)을 통해 정립한 프로세스 흐름은 다음과 같다.

```
Specification → Architecture → Implementation → Reflection → Review → Pull Request → Merge → Deployment
```

각 단계는 다음 단계로 넘어가기 전에 검증과 교정을 수행한다. Claude는 각 단계에서 최대 3번 반복을 시도한 후, 해결하지 못하면 사람의 개입을 요청한다.

### 2.3 핵심 구성 요소 (스킬)

**`/proceed` 스킬**
- Specification 이후의 SDLC 단계들을 자동화한다.
- 개발자가 요구사항 정의 후 반복적 단계를 직접 관리하지 않아도 되도록 한다.

**`/reflect` 스킬**
- Claude가 자신의 구현 결정을 자기 평가(self-evaluate)한다.
- 지름길(shortcut)을 식별하고 문서화하여 향후 개선의 기반을 만든다.

**멀티 에이전트 리뷰**
- 배포 전 추가 코드 품질 검증을 수행한다.
- 단일 에이전트의 맹점을 보완하는 역할을 한다.

**`/wrapup` 스킬**
- 레슨 러닝(lessons learned)을 로그로 남긴다.
- 브랜치를 관리하고 업데이트를 배포한다.

### 2.4 장기 프로젝트의 핵심 과제: 컴팩션(Compaction)

장기 실행 프로젝트에서는 "컴팩션" 현상이 발생한다. Claude가 워크플로우 단계를 잊어버려 일부 단계를 건너뛰는 문제다. 저자는 체크리스트 시스템을 도입하여 프로세스 상태 인식을 유지하는 방식으로 이 문제를 해결했다.

### 2.5 자기 개선 구조

"Claude는 시간이 지남에 따라 자기 개선한다." 문서화된 레슨을 통합하는 방식으로 Claude가 점진적으로 더 나은 결정을 내리도록 설계되어 있다.

## 3. 핵심 인사이트 정리

1. **AI 보조 개발에도 구조화된 프로세스가 필요하다**: 속도만 쫓는 바이브 코딩은 기술 부채와 품질 리스크를 낳는다. SDLC 프레임워크는 AI의 생산성을 유지하면서 품질을 통제할 수 있는 균형점을 제공한다.

2. **자기 반성(Reflection) 단계가 품질의 핵심이다**: Claude가 스스로 구현 결정을 평가하고 지름길을 문서화하는 `/reflect` 스킬은 단순 코드 생성을 넘어 "설명 가능한 AI 개발"을 가능하게 한다.

3. **멀티 에이전트 아키텍처는 단일 에이전트의 맹점을 보완한다**: 구현 에이전트와 리뷰 에이전트를 분리함으로써 동일한 AI가 자신의 실수를 간과하는 문제를 줄일 수 있다.

4. **컴팩션은 장기 AI 워크플로우의 실질적 위험이다**: 컨텍스트 손실로 인한 단계 누락은 AI 기반 개발에서 간과하기 쉬운 실패 모드다. 체크리스트 기반의 상태 관리가 현실적 해결책이다.

5. **3회 반복 후 인간 개입 요청 원칙**: Claude가 특정 문제를 3번 시도해도 해결하지 못하면 사람에게 넘기는 명시적 규칙은 AI의 무한 루프를 방지하고 인간-AI 협업의 경계를 명확히 한다.

6. **레슨 러닝의 자동화가 AI의 자기 개선을 만든다**: `/wrapup` 스킬을 통해 프로젝트별 레슨을 축적하면 Claude가 과거 실수를 반복하지 않는 구조가 형성된다. 이는 단순한 도구 사용을 넘어 "학습하는 개발 시스템"에 가깝다.

7. **프로덕션 환경으로의 확장은 추가 조정이 필요하다**: 저자 스스로 인정하듯, 현재 프레임워크는 개인 개발 환경에 최적화되어 있으며 실제 프로덕션 배포에는 추가적인 프로세스 조정이 요구된다.
