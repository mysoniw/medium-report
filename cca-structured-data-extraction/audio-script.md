# CCA 구조화된 데이터 추출 시나리오 - 오디오 스크립트

> 예상 분량: 약 12분 (3,800자)

---

CCA 딥다이브 시리즈 여섯 번째 에피소드입니다. 오늘은 가장 많은 수험생을 탈락시킨다는 구조화된 데이터 추출 시나리오입니다.

이 시나리오가 왜 이렇게 어려운 걸까요? 한마디로, "데모에서 작동하는 것"과 "프로덕션에서 작동하는 것"의 차이를 정밀하게 물어보기 때문입니다. Claude한테 "JSON으로 답해줘"라고 프롬프트에 적었더니 잘 나오죠? 근데 그게 진짜 신뢰할 수 있는 시스템인가요? 시험은 바로 그 지점을 파고듭니다.

## 3계층 신뢰성 모델 — 이것부터 외우세요

이번 시나리오의 핵심 프레임워크는 3계층 신뢰성 모델입니다.

레벨 1은 프롬프트 안내입니다. 시스템 프롬프트에 "Always return valid JSON"이라고 적는 거죠. 이걸 저자 Hightower는 "probabilistic nudge", 확률적 넛지라고 부릅니다. 행동경제학에서 넛지라는 개념 아시죠? 팔꿈치로 살짝 밀어주는 거요. 강제가 아닙니다. 살짝 밀어줄 뿐이에요. 그래서 대부분은 JSON이 나오지만, 가끔 마크다운 코드 펜스로 감싸서 나오거나, "Here is the extracted data"라는 설명 문구가 앞에 붙거나, 긴 문서에서는 닫는 중괄호가 잘리기도 합니다. 프롬프트만으로는 이런 실패를 막을 수 없어요.

레벨 2는 스키마 강제입니다. Anthropic SDK에서 tool_choice 파라미터를 사용해서 Claude를 특정 도구 호출로 강제하는 겁니다. 이걸 "tool-forcing"이라고 부르는데요. 실제로 아무 동작도 하지 않는 도구를 정의하고, 그 도구의 입력 스키마가 곧 우리가 원하는 JSON 스키마가 되는 겁니다. 이렇게 하면 마크다운 래퍼도, 설명 텍스트도 없이 스키마에 맞는 JSON만 나옵니다.

여기서 시험 함정이 하나 나옵니다. with_structured_output라는 메서드가 선택지에 등장하면 바로 제거하세요. 이건 LangChain 메서드입니다. "네이티브 Anthropic SDK로 구현하라"는 문제에서 이 선택지가 나오면, 저자의 표현을 빌리면 스타워즈의 애크바 제독이 말하죠 — "It's a trap!" 함정입니다. 정답은 tool_choice를 사용하는 방법이거나, client.messages.parse에 Pydantic 모델을 전달하는 방법입니다.

그런데 레벨 2만으로 충분할까요? 아닙니다. 여기서 저자가 아주 기막힌 비유를 씁니다. "A perfectly formatted check is not the same as a check that will clear." 완벽하게 포맷된 수표가 결제되는 수표와 같지 않다는 거예요. JSON 구조가 완벽해도, 그 안의 숫자가 맞는지는 전혀 별개의 문제라는 뜻입니다.

그래서 레벨 3, 프로그래밍적 시맨틱 검증이 필요합니다. 예를 들어 송장 데이터를 추출했는데, total 필드가 150이고 line items를 더하면 175라면? 스키마는 둘 다 유효한 숫자이니 통과시킵니다. 하지만 비즈니스 로직상 이건 틀린 거죠. 날짜 형식이 맞는지, 총액이 양수인지, 벤더 이름이 실제 존재하는 업체인지 — 이런 걸 코드로 검증해야 합니다.

시험은 항상 레벨 3 답을 보상합니다. 이거 정말 중요하니까 기억하세요.

## 자기보고 신뢰도 — 절대 믿지 마세요

여기서 또 하나 큰 함정이 나옵니다. "Claude한테 1에서 10 척도로 추출 신뢰도를 평가하라고 하면 되지 않나?" 아닙니다. Claude가 9점을 매겨도 데이터가 틀릴 수 있어요. 왜냐하면 환각으로 만들어낸 벤더 이름도, 모델에게는 정확한 값만큼이나 확신이 있거든요. 이게 LLM의 근본적인 한계입니다. 프로그래밍적 검증은 Claude에게 아무것도 묻지 않습니다. 외부 데이터, ground truth에 대해 독립적으로 확인하는 거예요.

## 재시도 루프 — 세 가지를 구분하세요

검증에서 실패하면 어떻게 할까요? 재시도를 해야겠죠. 그런데 시험에서는 재시도의 종류를 세 가지로 구분합니다.

첫 번째, blind retry. 그냥 "다시 해봐"라고만 하는 겁니다. 이건 항상 오답입니다.

두 번째, informed retry. "total은 150인데 line items 합계는 175야. 소스 문서를 다시 확인해봐"라고 구체적인 에러를 알려주는 겁니다. 저자가 이걸 디버깅에 비유하는데요, 컴파일러가 "47번 줄, 미선언 변수"라고 알려주는 것과 그냥 "무언가 잘못됐음"이라고 알려주는 것의 차이입니다. 당연히 전자가 문제 해결에 도움이 되겠죠.

세 번째, unbounded retry. "성공할 때까지 계속 재시도"입니다. 이것도 항상 오답이에요.

유일한 정답은 informed하면서 bounded, 즉 2~3회로 제한하고, 최대 횟수를 넘기면 인간에게 에스컬레이션하는 패턴입니다. 이 human-in-the-loop가 Context Management 도메인의 핵심 패턴이기도 합니다.

## 완전한 신뢰성 패턴

저자는 아주 영리한 교육 방식을 씁니다. 먼저 불완전한 코드를 보여주고 "어디가 잘못됐는지 찾아보라"고 합니다. 이 역방향 교수법이 시험 준비에 딱 맞는 거죠. 시험에서도 "겉보기에 괜찮은" 답을 식별해야 하니까요.

불완전한 코드는 이런 가정을 합니다 — API 호출이 항상 성공한다, 응답이 항상 비어있지 않다, 첫 번째 콘텐츠 블록이 항상 도구 사용 블록이다. 프로덕션에서 이 가정 중 하나라도 깨지면 시맨틱 검증에 도달하기도 전에 시스템이 크래시합니다.

완전한 패턴은 try/except로 API 실패, 스키마 실패, 응답 형태 이상 — 이런 hard failure를 잡고, 비즈니스 규칙 위반 같은 soft failure와 함께 하나의 재시도 루프에서 두 유형을 모두 처리합니다. 복합 실패 모드 문제에서는 모든 증상을 해결하는 답을 선택해야 합니다. 하나만 고치는 답은 함정이에요.

## 긴 문서와 대량 처리

마지막으로 두 가지 실전 포인트입니다.

50페이지 넘는 긴 문서에서는 "lost in the middle" 현상이 발생합니다. 문서 중간에 있는 정보의 추출 정확도가 떨어지는 거예요. 대응법은 문서를 청크로 나누고, 각 청크에서 독립적으로 추출한 다음, 결과를 병합하면서 중복을 제거하는 겁니다.

그리고 대량 처리 — 송장 5,000건을 추출해야 한다면 어떻게 하겠습니까? 하나씩 실시간 API 호출? 안티패턴입니다. Message Batches API를 써야 합니다. 비용도 50% 할인되고요.

## 시험에서 기억할 것

정리하겠습니다. 시험장에서 이것만 기억하세요.

하나, 프롬프트 기반 JSON 강제가 정답인 선택지는 제거하세요. 확률적 넛지일 뿐입니다.

둘, tool_choice 또는 client.messages.parse가 네이티브 SDK 패턴이고, with_structured_output는 LangChain입니다.

셋, LLM 자기보고 신뢰도는 프로덕션 검증이 될 수 없습니다.

넷, 재시도는 반드시 informed하고 bounded하며, 최대 횟수 이후 인간 에스컬레이션이 있어야 합니다.

다섯, hard failure와 soft failure 모두 처리하는 답이 완전한 답입니다.

이 시나리오가 탈락률이 높은 이유를 아시겠죠? 레벨 1에서 멈추는 게 너무 쉽기 때문입니다. "프롬프트에 JSON이라고 썼으니 됐지" — 이 유혹을 이겨내고 레벨 3까지 가야 합니다. 저자가 "Burn this into your memory"라고 했는데, 진짜 뇌에 새기셔야 합니다.

다음이자 마지막 에피소드에서는 개발자 생산성 시나리오를 다루겠습니다.
