# CCA 코드 생성 시나리오 — 오디오 스크립트

> 예상 분량: 약 12분 (3,800자)

---

CCA 딥다이브 시리즈 세 번째 에피소드입니다. 오늘 주제는 코드 생성 시나리오, Claude Code를 실제로 쓰는 개발자라면 가장 친숙하면서도 가장 함정이 많은 영역입니다.

자, 먼저 이 시나리오가 시험에서 차지하는 비중부터 짚고 갈게요. Claude Code Workflows 20퍼센트, Context Management 15퍼센트. 합치면 35퍼센트입니다. 시험 전체의 삼분의 일이 이 한 시나리오에 걸려 있어요. 절대 가볍게 볼 영역이 아닙니다.

## 컨텍스트 열화: 가장 위험한 직관

이 시나리오에서 가장 먼저, 그리고 가장 깊이 다뤄야 할 개념이 있습니다. 바로 context degradation, 컨텍스트 열화입니다.

여기서 "degradation"이라는 단어 선택에 주목해 주세요. overflow도 아니고 limit도 아닙니다. 열화예요. 용량이 넘쳐서 터지는 게 아니라, 품질이 서서히, 점진적으로 나빠진다는 뜻입니다. 이게 핵심이에요.

원문 저자 Rick Hightower는 이걸 attention dilution, 주의력 희석이라는 표현으로 설명합니다. 화학에서 용액을 희석하는 것처럼, 용매인 토큰이 많아지면 농도인 주의력이 낮아진다. 직관적이죠?

모델이 컨텍스트 윈도우를 처리할 때, 총 주의력은 고정되어 있습니다. 이걸 모든 토큰에 나눠줘야 해요. 컨텍스트가 커지면? 각 토큰이 받는 주의력 조각이 얇아집니다. 특히 중간에 있는 정보가 가장 얇은 주의력을 받아요. 이걸 업계에서 "lost in the middle" 효과라고 부릅니다.

저자는 이걸 이렇게 표현했어요. "The middle is where quality goes to die." 중간은 품질이 죽으러 가는 곳이다. 꽤 강렬한 의인화죠. 품질이 스스로 걸어가서 죽는 이미지입니다.

구체적인 실패 사례를 볼까요. 한 팀이 47개 서비스 파일, 토큰으로 38만 개를 단일 세션에 올려놓고 인증 미들웨어 리팩토링을 요청했습니다. 결과가 어떻게 됐냐면요. 처음 다섯 개 파일은 깔끔합니다. 중간으로 가면 변수 명명이 들쑥날쑥해지고 엣지 케이스가 빠져요. 마지막 파일들은 끝부분이라 주의력이 다시 올라가긴 하는데, 이번엔 처음 파일들과 import 패턴이 충돌합니다.

그래서 시험에 이런 함정이 나옵니다. "컨텍스트 윈도우를 늘리면 해결된다"는 선택지. 이건 항상 오답입니다. 윈도우를 늘리면 주의력을 더 얇게 분산시킬 뿐이에요. 200K 토큰의 집중된 컨텍스트가 1M 토큰을 덤프한 컨텍스트를 능가합니다.

## 정답 패턴: 분해하고, 집중하고, 합성하라

그러면 대규모 코드베이스 작업은 어떻게 해야 하느냐. 정답은 focused per-file passes, 파일별 집중 패스입니다.

여기서 "pass"라는 단어는 컴파일러의 패스에서 왔어요. 한 번에 하나의 대상만 체계적으로 처리한다는 뜻입니다.

워크플로우는 네 단계입니다. 첫째, 큰 작업을 파일이나 모듈 단위로 분해한다. 둘째, 관련 컨텍스트만 포함한 집중 세션에서 각 단위를 실행한다. 셋째, 결과를 합성한다. 넷째, 교차 파일 일관성을 검토한다. 명명 규칙, import 경로, 인터페이스 계약 같은 것들이요.

저자는 수술 비유를 씁니다. "Think of it like surgery." 외과의사는 모든 장기를 동시에 수술하지 않는다. 한 영역에 집중하고, 완전한 주의력으로 작업하고, 다음으로 넘어간다. 의료 비유를 통해 범위가 곧 품질이라는 메시지를 생명과 직결시킨 거죠.

시험에서 "전체 src 디렉토리를 로드하고 리팩토링"이 선택지에 나오면, 그건 항상 오답입니다.

## CLAUDE.md 계층: 프로젝트 vs 사용자

다음으로 중요한 출제 영역이 CLAUDE.md의 계층 구조입니다.

두 가지 레벨이 있어요. 프로젝트 레벨은 프로젝트 안의 .claude 폴더 밑에 둡니다. Git에 커밋되니까 팀 전체가 자동으로 공유받아요. 여기에는 팀 코딩 표준, 아키텍처 규칙 같은 것들이 들어갑니다.

사용자 레벨은 홈 디렉토리의 .claude 폴더 밑에 둡니다. Git에 안 들어가요. 개인 선호도, 에디터 설정 같은 것들이죠.

시험 함정이 이렇게 나옵니다. "일부 개발자만 코딩 표준을 따르고 다른 개발자는 따르지 않는다. 어떻게 해결하나?" 선택지 중에 "각 개발자의 사용자 레벨 CLAUDE.md에 표준을 넣는다"가 있으면 그건 오답이에요. 정답은 항상 프로젝트 레벨입니다.

저자가 CLAUDE.md를 이렇게 묘사합니다. "A tech lead who is always present, never forgets a rule, and never skips a review." 항상 존재하고, 규칙을 잊지 않고, 리뷰를 건너뛰지 않는 테크 리드. 프로그래밍적 접근의 장점을 이상적인 동료의 이미지로 번역한 거예요.

핵심은 이겁니다. CLAUDE.md는 컨텍스트 엔지니어링이다. 매 세션에 자동으로 주입되니까, 매번 프롬프트에 규칙을 직접 타이핑하는 것보다 한 번 잘 작성해두는 게 압도적으로 효율적입니다.

## CI/CD 통합: -p 플래그의 함정

실무에서도 시험에서도 놓치기 쉬운 부분이 CI/CD 통합입니다.

-p 플래그, 풀네임으로는 --print 플래그 없이 Claude Code를 CI 파이프라인에서 실행하면 어떻게 될까요? 에러가 나거나 느려지는 게 아닙니다. 파이프라인이 영원히 멈춥니다. 왜냐하면 인터랙티브 UI가 사용자 입력을 기다리는데, CI 환경에는 입력을 줄 사용자가 없으니까요.

함께 써야 하는 게 --bare 플래그입니다. 훅, LSP, 스킬 로딩, 자동 메모리 같은 자동 발견 기능을 전부 스킵해서 재현 가능한 동작을 보장해요. 구조화된 출력이 필요하면 --output-format json도 추가하고요.

그리고 Batch API 관련 문제도 나옵니다. 개발자가 머지를 기다리고 있는 상황이면 Real-Time API를 쓰고, "nightly"나 "weekly" 같은 스케줄 작업이면 Batch API를 쓴다. Batch는 50퍼센트 할인이 있거든요. 시험에서 키워드로 구분하면 됩니다.

## 커스텀 스킬과 코디네이터 패턴

마지막 두 가지 핵심 토픽입니다.

커스텀 스킬은 간단합니다. 워크플로우가 세 단계 이상이고 두 번 이상 반복되면, 스킬로 인코딩하라. 주의할 점은 스킬이 CLAUDE.md를 참조해야지, CLAUDE.md의 내용을 복사해서 넣으면 안 된다는 겁니다. 이중 유지보수의 악몽이 시작되거든요.

코디네이터 패턴은 좀 더 고급입니다. 코디네이터 세션이 전체 구조를 가볍게 읽고, 작업 계획을 세우고, 각 파일에 필요한 컨텍스트만 담은 집중 세션을 할당하고, 마지막에 모든 결과의 일관성을 검토합니다.

저자는 이걸 context forking, 컨텍스트 포킹이라고 부릅니다. Unix의 fork 시스템 호출에서 따온 거예요. 부모 프로세스의 컨텍스트 중 필요한 부분만 자식 프로세스에 복제한다는 의미죠. 이건 다음 에피소드에서 다룰 허브앤스포크 아키텍처와 직접 연결됩니다.

## 시험 전략 요약

정리하겠습니다. 코드 생성 시나리오에서 기억해야 할 다섯 가지.

하나, "컨텍스트 윈도우를 늘리자"는 선택지는 항상 오답이다.
둘, 대규모 리팩토링의 정답은 분해, 집중, 합성, 검토다.
셋, 팀 표준은 반드시 프로젝트 레벨 CLAUDE.md에 둔다.
넷, CI/CD에서 -p 플래그 없이 실행하면 파이프라인이 영원히 멈춘다.
다섯, 반복 워크플로우는 스킬로, 대규모 작업은 코디네이터 패턴으로 확장한다.

저자 Hightower의 어투를 잠깐 짚자면, 이 아티클에서는 이전 에피소드의 전투 브리핑 톤과는 달리 "함께 분석하는" 페어 프로그래밍 톤을 채택했어요. "Let's get into it", "Walk through what actually happens" 같은 표현으로 독자를 워크스루에 초대하는 느낌입니다. 코드 생성이라는 개발자에게 친숙한 주제에 맞춘 선택이죠. 하지만 안티패턴 경고에서는 여전히 "That is the trap"처럼 시험 전투 프레이밍이 살아 있습니다.

다음 에피소드에서는 시리즈의 하이라이트, 멀티에이전트 리서치 시나리오를 다루겠습니다.
