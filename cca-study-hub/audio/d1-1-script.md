CCA Foundations 시험 대비, 도메인 원 에이전틱 아키텍처 파트 원입니다. 이 도메인은 시험 전체에서 가장 높은 비중인 27퍼센트를 차지하며, 60문항 중 약 16문항이 여기서 출제됩니다. 합격 점수가 천 점 만점에 720점이라는 점을 고려하면, 이 도메인 하나를 확실히 잡는 것만으로도 합격에 상당히 유리해집니다. 오늘은 이 도메인의 전반부, 핵심 아키텍처 패턴과 워크플로우를 다루겠습니다.

먼저, 이 도메인을 관통하는 세 가지 대원칙을 기억해주십시오. 첫째, 프롬프트는 가이던스일 뿐이고, 코드가 법이다. 영어로 "Prompts are guidance, Code is law"라고 합니다. 비즈니스 규칙을 시스템 프롬프트에 적어놓는 것은 희망사항일 뿐이고, 반드시 코드로 강제해야 합니다. 둘째, 서브에이전트는 코디네이터의 컨텍스트를 상속하지 않습니다. 각 서브에이전트는 블랭크 슬레이트(blank slate), 즉 완전히 빈 상태에서 시작합니다. 셋째, 결정론적 규칙이 모델의 판단보다 항상 우선합니다. 에스컬레이션이든 라우팅이든 검증이든, 결정론적 규칙(deterministic rules)이 정답입니다.

이제 첫 번째 핵심 개념인 에이전틱 루프(Agentic Loop)를 살펴보겠습니다. 에이전틱 루프는 Claude 기반 에이전트의 가장 기본적인 실행 흐름입니다. 사용자가 입력을 보내면 Claude가 응답을 생성하고, 만약 도구 호출이 필요하면 스탑 리즌(stop reason)으로 tool_use를 반환합니다. 그러면 애플리케이션이 해당 도구를 실행하고 그 결과를 tool_result로 다시 Claude에게 전달합니다. Claude는 이 결과를 보고 다음 단계를 결정하며, 이 과정은 end_turn이라는 스탑 리즌이 반환될 때까지 반복됩니다. 여기서 시험에 나올 수 있는 세 번째 스탑 리즌이 max_tokens인데, 이것은 출력이 토큰 한도에 도달해서 잘렸을 수 있다는 의미이므로 후처리가 필요합니다. 이 세 가지 스탑 리즌은 반드시 구분할 수 있어야 합니다. tool_use가 반환되면 tool_result를 반드시 돌려보내야 하고, end_turn이면 사용자에게 최종 응답을 전달하며, max_tokens이면 출력이 잘렸을 가능성을 고려해 적절한 후처리를 해야 합니다.

다음으로 살펴볼 개념은 이 시험에서 가장 많이 출제되는 단일 패턴인 코디네이터 서브에이전트(Coordinator-Subagent) 패턴입니다. 중앙 코디네이터가 작업을 전문화된 서브에이전트들에게 위임하고, 각 서브에이전트의 결과를 수집해서 종합하는 구조입니다. 뉴스룸에 비유하면, 편집자가 코디네이터이고 정치 기자, 경제 기자, 탐사 기자가 각각의 서브에이전트입니다. 편집자가 기사를 배정하면 기자들이 각자 취재한 뒤 편집자에게 제출합니다. 여기서 중요한 것은 기자끼리 직접 기사를 공유하지 않는다는 점입니다. 예를 들어 리서치 시스템에서는 코디네이터 아래에 웹 리서처, 문서 분석기, 데이터 추출기, 팩트 체커 이렇게 네 개의 서브에이전트가 있고, 각각 네다섯 개의 전문 도구를 갖습니다.

여기서 중요한 것은 토폴로지, 즉 통신 구조입니다. 시험에서 정답으로 취급되는 것은 허브앤스포크(Hub-and-Spoke) 패턴입니다. 코디네이터가 허브이고 서브에이전트들이 스포크입니다. 모든 통신은 허브를 경유해야 하며, 스포크 간 직접 통신은 금지됩니다. 반대로 풀 메시(Full Mesh) 토폴로지는 모든 에이전트가 서로 직접 통신하는 구조인데, 이것은 CCA 시험에서 안티패턴으로 취급됩니다. 시험에서 "서브에이전트 A가 서브에이전트 B에 직접 결과를 전달한다"라는 선택지가 보이면 항상 오답입니다. 허브앤스포크의 복잡도는 O(n)인 반면 풀 메시는 O(n 제곱)이고, 디버깅 측면에서도 허브에서 모든 흐름을 추적할 수 있는 허브앤스포크가 압도적으로 유리합니다. 항공 허브에 비유하면, 인천공항이 허브이고 각 도시가 스포크입니다. 모든 항공편이 인천을 경유하지, 부산에서 제주로 가는 길에 대구를 거치지 않는 것과 같은 원리입니다.

다섯 가지 패턴에 들어가기 전에, 증강된 LLM(Augmented LLM)이라는 개념을 짚고 넘어가겠습니다. 증강된 LLM은 순수 LLM에 도구(tools), 검색(retrieval), 메모리(memory)를 결합한 것으로, 에이전틱 시스템의 최소 빌딩 블록입니다. 멀티에이전트 시스템의 각 에이전트는 결국 전문화된 증강 LLM이라고 이해하시면 됩니다. 이제 Anthropic이 공식적으로 정의한 다섯 가지 워크플로우 패턴을 하나씩 살펴보겠습니다. 첫 번째는 프롬프트 체이닝(Prompt Chaining)입니다. 한 LLM 호출의 출력이 다음 호출의 입력이 되는 선형 파이프라인 구조이며, 각 단계 사이에 게이트(gate)를 추가하여 품질을 검증할 수 있습니다. 이메일을 작성하고 톤을 검토한 뒤 발송하는 것이 대표적인 예입니다. 두 번째는 라우팅(Routing) 패턴입니다. 입력을 먼저 분류하고 그 유형에 따라 전문 핸들러로 분기하는 구조입니다. 고객 문의를 기술, 결제, 배송으로 나누어 각각의 전문 처리기로 보내는 것이 예입니다. 세 번째는 패럴라이제이션(Parallelization)인데, 여기에는 두 가지 하위 유형이 있습니다. 섹셔닝(Sectioning)은 독립적인 서브태스크를 병렬로 실행하는 것이고, 보팅(Voting)은 동일한 태스크를 여러 번 실행해서 다양한 결과를 합성하는 것입니다. 세 개의 소스에서 동시에 검색하는 것이 섹셔닝이고, 세 명의 리뷰어가 같은 코드를 독립적으로 평가하는 것이 보팅입니다. 네 번째는 오케스트레이터 워커(Orchestrator-Workers) 패턴입니다. 중앙 LLM이 태스크를 동적으로 분해해서 워커들에게 위임하는 구조로, 앞서 설명한 코디네이터 서브에이전트 패턴과 본질적으로 같은 개념입니다. 다섯 번째는 이밸류에이터 옵티마이저(Evaluator-Optimizer) 패턴입니다. 한 LLM이 콘텐츠를 생성하고 다른 LLM이 평가하여 피드백을 주는 루프 구조입니다. 카피라이터 LLM이 마케팅 문구를 작성하고 편집자 LLM이 브랜드 가이드라인 준수 여부를 체크하여 수정을 요구하는 것이 대표적인 예입니다. 시험에서는 이 다섯 패턴의 시나리오를 주고 어떤 패턴에 해당하는지 식별하도록 요구하므로, 각 패턴의 핵심 특징과 적합한 사용 사례를 확실히 구분해두어야 합니다.

마지막으로 워크플로우(Workflow)와 에이전트(Agent)의 구분을 짚겠습니다. Anthropic은 이 둘을 명확히 구분합니다. 워크플로우는 개발자가 작성한 미리 정의된 코드 경로(predefined code paths)로 LLM을 오케스트레이션하는 것입니다. 예측 가능성이 높고 디버깅이 쉽습니다. 반면 에이전트는 LLM이 자신의 프로세스와 도구 사용을 동적으로 결정(dynamically direct)하는 것입니다. 유연하지만 예측이 어렵습니다. 시험에서 자주 출제되는 포인트는 "언제 워크플로우를 쓰고 언제 에이전트를 쓰는가"입니다. 원칙은 간단합니다. 예측 가능한 고정 프로세스에는 워크플로우, 어떤 도구를 어떤 순서로 사용할지 미리 정할 수 없는 복잡한 상황에만 에이전트를 사용합니다. 파트 원은 여기까지입니다. 파트 투에서는 에스컬레이션, 컨텍스트 격리, 안티패턴, 그리고 암기 필수 숫자들을 다루겠습니다.
