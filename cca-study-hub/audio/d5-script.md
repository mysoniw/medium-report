# CCA Domain 5 컨텍스트 관리 오디오 스크립트

> 예상 분량: 약 16분 (약 3,300자)

---

CCA 시험 대비, 도메인 파이브 컨텍스트 관리와 신뢰성입니다. 비중은 15퍼센트, 60문항 중 약 9문항이 출제되며, 다섯 개 도메인 중 가장 가중치가 낮습니다. 하지만 절대 가볍게 보면 안 됩니다. 컨텍스트 관리는 아키텍처, 도구 설계, 프롬프트 엔지니어링 등 나머지 네 도메인을 관통하는 횡단 개념입니다. 이걸 모르면 다른 도메인 문제도 틀릴 수 있다는 뜻이죠.

이 도메인의 핵심 질문은 하나입니다. Claude의 컨텍스트 윈도우를 어떻게 효율적으로 관리하고, 토큰 비용을 최적화하며, 긴 대화에서도 품질을 유지하느냐.

먼저 Context Degradation, 컨텍스트 열화부터 시작합니다. 이것은 도메인 파이브의 가장 기본적인 개념이자, 시험에서 가장 자주 나오는 주제입니다. 핵심을 한 문장으로 요약하면, 컨텍스트 열화는 용량 문제가 아니라 주의력 문제입니다. Transformer 모델의 총 attention은 고정되어 있습니다. 컨텍스트에 토큰이 많아질수록 각 토큰에 할당되는 주의력은 얇아집니다. 이걸 attention dilution, 주의력 희석이라고 부릅니다.

여기서 Lost in the Middle 효과를 꼭 기억하세요. 컨텍스트 시작 부분은 주의력이 높고, 끝부분도 주의력이 회복되지만, 중간부는 주의력이 가장 낮습니다. 실제로 47개 파일 380K 토큰을 단일 세션에 로드한 사례에서, 초반 파일은 깔끔했고, 중간 파일은 변수명 불일치와 엣지 케이스 누락이 발생했으며, 후반 파일은 초반과 import 패턴이 충돌했습니다. 시험에서 "컨텍스트 윈도우를 늘리면 해결된다"는 선택지가 나오면 즉시 제거하세요. 항상 오답입니다. 200K 토큰의 집중된 컨텍스트가 1M 토큰의 덤프된 컨텍스트보다 성능이 좋습니다.

그렇다면 대규모 작업의 정답 패턴은 무엇일까요. Focused Per-File Passes, 파일별 집중 패스입니다. 네 단계로 구성됩니다. 첫째 Decompose, 대규모 태스크를 파일이나 모듈 단위로 분해합니다. 둘째 Execute, 각 단위를 필요한 컨텍스트만 담은 집중 세션에서 실행합니다. 셋째 Compose, 결과를 합성합니다. 넷째 Review, 교차 파일 일관성을 검토합니다. 이 패턴의 고급 구현이 Coordinator Pattern입니다. 코디네이터 세션이 전체 구조를 가볍게 읽고, 작업 계획을 수립한 뒤, 각 파일에 집중 세션을 할당하고, 마지막에 교차 파일 일관성을 검토합니다.

다음으로 Prompt Caching입니다. 암기해야 할 숫자가 많으니 집중하세요. 캐싱이 가능하려면 최소 1024 토큰 이상이어야 합니다. 기본 TTL은 5분이고, 확장 TTL은 1시간이며 추가 비용이 듭니다. 캐시 읽기 비용은 기본의 0.1배, 즉 90퍼센트 절감입니다. 하지만 캐시 쓰기 비용은 5분 TTL 기준 1.25배로 오히려 25퍼센트 증가하고, 1시간 TTL은 2.0배로 100퍼센트 증가합니다. 시험 함정이 있습니다. "Prompt Caching은 모든 토큰 비용을 90퍼센트 절감한다"는 오답입니다. 90퍼센트 절감은 캐시 읽기 토큰에만 해당하고, 쓰기는 비용이 증가합니다.

Token Economics 비교도 중요합니다. Standard API는 비용 1.0배 기준, 실시간 처리, ZDR 호환입니다. Batch API는 0.5배로 50퍼센트 절감되지만, 처리에 최대 24시간이 걸리고, ZDR을 지원하지 않습니다. Prompt Caching은 읽기 0.1배, 실시간 호환, ZDR 호환입니다. 판단 기준은 간단합니다. 누가 기다리고 있는가. 사람이 기다리면 Real-Time API가 유일한 선택입니다. 아무도 기다리지 않는 백그라운드 작업이고 ZDR이 불필요하면 Batch API를 씁니다. 규제 산업에서 ZDR이 필요하면 Batch API는 컴플라이언스 위반이므로 즉시 제거하세요.

Context Isolation, 컨텍스트 격리입니다. 서브에이전트는 부모 에이전트의 컨텍스트를 자동으로 상속하지 않습니다. 빈 컨텍스트에서 시작합니다. 직관에 반하기 때문에 시험에서 자주 출제됩니다. Context Forking이라고 부르며, Unix fork에서 차용한 개념으로, 코디네이터가 서브에이전트에게 필요한 정보만 선택적으로 전달해야 합니다.

Retry 전략으로 넘어갑니다. 네 가지 유형을 구분하세요. Blind retry는 에러 피드백 없이 그냥 다시 시도하는 것으로 항상 오답입니다. Unbounded retry, 성공할 때까지 무한 재시도도 항상 오답입니다. Informed retry는 구체적인 에러 피드백을 포함하여 재시도하는 것으로 정답 패턴입니다. Content-modified retry는 에러 피드백을 컨텍스트에 추가하여 재시도하는 것으로 역시 정답 패턴입니다. 정답 공식을 외우세요. informed 플러스 bounded 2에서 3회 플러스 human escalation.

Write-Select-Compress 3단계 전략입니다. Anthropic이 공식적으로 제시하는 컨텍스트 관리 프레임워크입니다. Write 단계에서 시스템 프롬프트, 도구 정의, 정적 문서 등 컨텍스트에 넣을 정보를 사전에 준비합니다. Select 단계에서 현재 태스크에 관련 있는 정보만 동적으로 선택합니다. RAG 벡터 검색이 대표적인 Select 구현입니다. Compress 단계에서 불필요한 정보를 제거하고 핵심만 유지합니다.

Tool Result Summarization, 도구 결과 요약도 중요합니다. 15,000 토큰짜리 API 응답을 그대로 컨텍스트에 넣는 것은 안티패턴입니다. 핵심 정보만 추출하여 요약본을 삽입해야 합니다. 원본 전체 삽입은 주의력 희석을 유발합니다.

대화 이력 관리의 정답 패턴은 하이브리드입니다. 슬라이딩 윈도우로 최근 메시지 원문을 유지하고, 오래된 메시지는 요약으로 대체합니다. 중요 정보는 컨텍스트 시작과 끝에 배치하여 Lost in the Middle에 대응합니다.

안티패턴 다섯 개를 정리합니다. 첫째, 윈도우를 늘려 모든 파일을 로드하는 것. 둘째, 실시간 워크플로우에 Batch API를 적용하는 것. 셋째, 대규모 파일을 단일 세션에 전부 로드하는 것. 넷째, 에스컬레이션 시 전체 대화 기록을 그대로 전달하는 것. 다섯째, 모든 토큰이 90퍼센트 절감된다고 오해하는 것.

마지막으로 암기 숫자와 시험 팁입니다. Prompt Caching 최소 토큰 1024, 기본 TTL 5분, 확장 TTL 1시간, 캐시 읽기 0.1배, 캐시 쓰기 5분 1.25배, 캐시 쓰기 1시간 2.0배, Batch API 50퍼센트 절감, Batch 처리 윈도우 최대 24시간, 재시도 제한 2에서 3회, 에이전트당 권장 도구 수 4에서 5개. 그리고 시험에서 "increase window", "load everything", "larger context"라는 문구가 보이면 즉시 제거, "decompose", "focused session", "per-file"이 보이면 정답 후보입니다.

도메인 파이브 컨텍스트 관리, 비중은 작지만 다른 도메인을 관통하는 핵심입니다. 숫자를 외우고, 안티패턴을 식별하는 연습을 하면 9문항을 확실히 잡을 수 있습니다.
