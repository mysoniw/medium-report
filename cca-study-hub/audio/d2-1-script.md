CCA 시험 대비, 도메인 투. 도구 설계와 엠시피 통합입니다. 이 도메인은 전체 시험의 십팔 퍼센트를 차지하며, 예상 출제 문항은 약 열한 문제입니다. 가중치만 놓고 보면 도메인 원이나 도메인 포보다 낮아 보이지만, 초기 응시자 피드백에 따르면 이 도메인에서 예상 외 실점이 가장 많이 발생합니다. 그래서 다크호스 도메인이라고 부릅니다. 특히 엠시피의 툴과 리소스 경계를 정확히 구분하지 못해서 떨어지는 수험생이 많으니, 여섯에서 여덟 시간 정도의 학습 시간을 투자하시길 권합니다.

이 도메인이 테스트하는 핵심 역량은 세 가지입니다. 클로드가 도구를 선택하고 실행하는 메커니즘, 엠시피(MCP) 프로토콜의 구조적 이해, 그리고 도구 설계의 모범 사례와 안티패턴 식별입니다.

먼저 엠시피의 세 가지 프리미티브(Primitives)부터 살펴보겠습니다. 엠시피는 에이전트가 외부 세계와 상호작용하는 방식을 세 가지로 정의합니다. 툴(Tool), 리소스(Resource), 프롬프트(Prompt)입니다.

툴은 동사에 해당합니다. 실행 가능한 함수로서, 데이터베이스 쿼리 실행, 에이피아이 호출, 파일 쓰기처럼 시스템에 변화를 일으키는 작업을 담당합니다. 판별 기준은 간단합니다. "클로드가 무언가를 실행해서 변화를 일으켜야 하는가?" 그렇다면 툴입니다.

리소스는 명사에 해당합니다. 읽기 전용 데이터로서, 문서, 스키마, 지식 베이스처럼 클로드가 맥락을 파악하기 위해 참조만 하면 되는 것들입니다. "클로드가 이것을 읽기만 하면 되는가?" 그렇다면 리소스입니다.

프롬프트는 패턴에 해당합니다. 재사용 가능한 템플릿이나 워크플로우로, 코드 리뷰 템플릿이나 보고서 생성 패턴처럼 반복적으로 사용되는 작업 패턴을 정의합니다.

시험에서 가장 빈번하게 출제되는 함정이 바로 이 툴과 리소스의 경계입니다. 예를 들어 "고객 프로필 조회"가 나왔을 때, 데이터베이스에서 능동적으로 쿼리를 실행해서 조회하는 것이라면 이것은 툴입니다. 반면에 이미 로드된 데이터를 참조하는 것이라면 리소스입니다. 같은 "조회"라는 단어도 실행이 수반되는지, 단순 읽기인지에 따라 분류가 달라집니다. 암기법으로는 "동사, 명사, 패턴"을 기억하세요. 툴즈 이퀄즈 버브즈, 리소시즈 이퀄즈 나운즈, 프롬프츠 이퀄즈 패턴즈입니다.

다음으로 엠시피의 호스트, 클라이언트, 서버 삼계층 아키텍처를 설명하겠습니다. 방금 배운 세 가지 프리미티브가 "무엇을 제공하는가"라면, 이 삼계층은 "어떤 구조로 통신하는가"에 대한 이야기입니다.

호스트(Host)는 사용자가 직접 대면하는 앱입니다. 클로드 데스크톱이나 클로드 코드가 여기에 해당합니다. 호스트는 여러 클라이언트를 관리하고, 보안 정책과 사용자 동의를 제어합니다.

클라이언트(Client)는 호스트 안에서 엠시피 서버와 통신하는 프로토콜 커넥터입니다. 중요한 것은 클라이언트가 독립적인 앱이 아니라 호스트 내부의 커넥터라는 점입니다. 시험에서 "클라이언트는 독립 앱이다"라는 선지가 나오면 그것은 오답입니다.

서버(Server)는 도구, 리소스, 프롬프트를 제공하는 외부 프로세스입니다. 깃허브 엠시피 서버, 지라 엠시피 서버처럼 특정 기능에 집중합니다.

이 세 계층의 관계를 숫자로 기억하세요. 호스트 일 대 클라이언트 엔, 각 클라이언트 일 대 서버 일. 즉, 하나의 호스트가 여러 클라이언트를 관리하지만, 각 클라이언트는 정확히 하나의 서버에만 연결됩니다. 하나의 클라이언트가 여러 서버에 연결된다는 선지는 항상 오답입니다.

이제 도구 설명이 곧 라우팅이라는 핵심 개념을 다루겠습니다. 클로드는 도구를 이름이 아닌 설명, 즉 디스크립션(description)으로 선택합니다. 도구의 설명이 곧 라우팅 테이블인 셈입니다. 이것을 영어로 "툴 디스크립션 애즈 라우팅 메커니즘"이라고 합니다.

좋은 도구 설명에는 네 가지 요소가 포함됩니다. 첫째, 긍정 경계(Positive bounds). 이 도구가 하는 것을 명시합니다. 둘째, 부정 경계(Negative bounds). 이 도구가 하지 않는 것을 명시합니다. 셋째, 입력 명세. 넷째, 출력 명세입니다. 예를 들어 "파이썬 소스 파일에 정적 분석을 수행한다, 코드를 실행하거나 파일을 수정하지 않는다"처럼, 긍정과 부정 경계를 모두 갖춘 설명이 정확한 라우팅을 가능하게 합니다. 반대로 "뭔가를 분석한다"같은 모호한 설명은 미스라우팅(misrouting)을 유발합니다.

여기서 핵심적인 개념이 디스크립션-디서먼트 루프(Description-Discernment Loop)입니다. 도구 설명을 한 번 작성하고 끝내는 것이 아니라, 반복적 피드백 루프로 개선해 나가는 것이 정답 패턴입니다. 흐름은 이렇습니다. 도구 설명을 작성하면 클로드가 그 설명에 기반해 도구를 선택하고 출력을 생성합니다. 그 결과를 코드 리뷰와 테스트로 판별(Discernment)합니다. 판별 결과에 따라 도구 설명을 업데이트하고, 이 과정을 반복합니다. 시험에서 "프롬프트를 한 번 수정한다"는 답과 "피드백 루프로 지속 개선한다"는 답이 나오면, 항상 루프가 정답입니다.

마지막으로 도구 수 임계값, 즉 포-파이브 툴 프린시플(4-5 Tool Principle)을 설명하겠습니다. 에이전트당 최적 도구 수는 사에서 오 개입니다. 이를 초과하면 어텐션 택스(attention tax), 즉 주의력 세금이 발생하여 도구 선택 정확도가 떨어집니다.

구간별로 정리하면, 사에서 오 개는 최적 상태로 도구 설명 구체화만으로 미세 조정이 가능합니다. 육에서 십 개는 주의 구간으로 설명 개선으로 해결 가능하지만 분할을 검토해야 합니다. 십이 개 이상은 분할 필수입니다. 코디네이터-서브에이전트 패턴으로 분리해야 합니다. 십오 개 이상은 스위스 아미 에이전트(Swiss Army Agent) 또는 슈퍼 에이전트(Super Agent)라 불리는 대표적 안티패턴입니다.

여기서 반드시 기억해야 할 것은 이 제한이 에스디케이(SDK)의 기술적 제약이 아니라 아키텍처 설계 원칙이라는 점입니다. "에스디케이가 도구 수를 제한한다"는 선지는 항상 오답입니다. 분할이 정답인 이유는 두 가지입니다. 첫째, 도구 선택 정확도가 저하됩니다. 십오 개 이상의 도구가 있으면 클로드가 매 턴마다 더 많은 옵션을 평가해야 해서 오선택 확률이 증가합니다. 둘째, 최소 권한 원칙(Principle of Least Privilege) 위반입니다. 고객 지원 에이전트가 에이치알 정책이나 마케팅 데이터에 접근 가능하면 보안과 컴플라이언스 리스크가 생깁니다. 이 두 문제를 동시에 지적하는 답이 시험에서 정답이 되는 경우가 많습니다.

파트 원 정리입니다. 엠시피 프리미티브 세 가지는 동사인 툴, 명사인 리소스, 패턴인 프롬프트. 아키텍처 삼계층은 호스트, 클라이언트, 서버이며 호스트 일 대 클라이언트 엔, 클라이언트 일 대 서버 일의 관계. 도구 설명이 곧 라우팅이며 좋은 설명에는 긍정 경계, 부정 경계, 입출력 명세가 포함. 디스크립션-디서먼트 루프는 일회성이 아닌 반복적 개선 패턴. 에이전트당 사에서 오 개 도구가 최적이며, 십이 개 이상이면 반드시 서브에이전트로 분할. 이것은 에스디케이 제한이 아닌 아키텍처 원칙입니다. 파트 투에서 엠시피 스코핑, 샌드박싱, 그리고 안티패턴을 이어서 다루겠습니다.
