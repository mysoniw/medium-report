CCA 시험 대비 도메인 투, 파트 투입니다. 파트 원에서 엠시피 프리미티브 세 가지, 삼계층 아키텍처, 도구 설명 라우팅, 그리고 도구 수 임계값을 다뤘습니다. 이번 파트에서는 엠시피 스코핑, 도구 샌드박싱, 그리고 시험에서 자주 출제되는 고급 개념들과 안티패턴을 정리하겠습니다.

먼저 엠시피 스코핑(MCP Scoping)입니다. 엠시피 설정은 클로드엠디(CLAUDE.md)와 동일한 계층 원칙을 따릅니다. 핵심 원칙은 "정의는 팀에, 비밀은 나에게"입니다. 영어로 "디피니션즈 투 팀, 시크릿츠 투 미"라고 외우시면 됩니다.

두 개의 파일을 구분해야 합니다. 첫째, 닷엠시피닷제이슨(.mcp.json)은 프로젝트 루트에 위치하며, 팀 공유 엠시피 서버 정의를 담습니다. 이 파일은 버전 관리 시스템에 포함됩니다. 둘째, 틸다 슬래시 닷클로드닷제이슨(~/.claude.json)은 홈 디렉토리에 위치하며, 개인 엠시피 서버와 크레덴셜을 담습니다. 이 파일은 버전 관리에 포함되지 않습니다.

의사결정 흐름은 이렇습니다. 팀 전체가 이 서버를 사용하는가? 아니라면 개인 설정 파일에 전부 정의합니다. 예라면, 개인 크레덴셜이 필요한가? 아니라면 프로젝트 설정 파일에 전부 정의합니다. 크레덴셜이 필요하다면, 서버 정의는 프로젝트 설정 파일에, 크레덴셜은 개인 설정 파일이나 환경변수에 분리합니다. 시험 함정으로, 닷엠시피닷제이슨에 에이피아이 토큰을 직접 넣으면 버전 관리에 크레덴셜이 노출되어 보안 위반입니다. 반대로 모든 것을 개인 설정 파일에 넣으면 팀원이 공유 서버에 접근할 수 없습니다.

다음은 도구 샌드박싱(Tool Sandboxing)입니다. 도구에 최소 권한 원칙, 즉 프린시플 오브 리스트 프리빌리지(Principle of Least Privilege)를 적용하는 것입니다. 에이전트가 필요 이상의 도구에 접근하면 보안 리스크가 발생합니다.

샌드박싱은 네 가지 영역입니다. 네트워크 제한으로 승인된 에이피아이만 호출하게 하고, 파일시스템 제한으로 필요한 디렉토리만 접근하게 하고, 도구 범위 제한으로 역할에 맞는 도구만 부여하고, 실행 권한 제한으로 포스트툴유즈(PostToolUse) 훅을 통해 사후 검증을 수행합니다.

시험 출제 패턴을 하나 알려드리겠습니다. "고객 지원 에이전트에 십오 개 도구가 있고 에이치알 정책 조회도 포함되어 있다"는 시나리오가 나오면, 보안 리스크와 도구 선택 정확도 저하 두 가지 문제를 동시에 지적하는 답이 정답입니다.

이제 케이퍼빌리티 네고시에이션(Capability Negotiation)입니다. 연결 초기화 시 클라이언트와 서버가 지원 기능을 교환합니다. 서버가 툴즈 케이퍼빌리티만 선언하면 클라이언트는 리소스를 요청하지 않습니다. 서버가 지원하지 않는 기능을 요청하면 프로토콜 오류가 발생합니다.

다음은 샘플링(Sampling)입니다. 엠시피 서버가 클라이언트를 통해 엘엘엠에 완성 요청을 하는 역방향 채널입니다. 핵심은 서버가 직접 모델에 접근하지 않고 반드시 클라이언트와 호스트를 경유한다는 점입니다. 이는 휴먼 오버사이트(human oversight)를 유지하기 위해서입니다. 시험에서 "엠시피 서버가 엘엘엠을 직접 호출한다"는 선지는 항상 오답입니다.

병렬 도구 사용(Parallel Tool Use)입니다. 클로드는 한 번의 응답에서 여러 도구를 동시에 요청할 수 있습니다. 디세이블 패러렐 툴 유즈(disable_parallel_tool_use) 파라미터를 트루로 설정하면 순차 실행, 기본값인 폴스에서는 독립적인 도구들이 동시 호출됩니다.

이즈에러(is_error) 필드도 중요합니다. 도구 실행이 실패했을 때 툴 리절트에 이즈에러 트루를 포함하면, 클로드가 에러 상황을 이해하고 재시도하거나 대안 도구를 선택하거나 사용자에게 설명하는 등 적절한 대응을 할 수 있습니다. 이즈에러를 설정하지 않고 에러 메시지만 반환하면, 클로드가 이것을 정상 결과로 해석할 수 있습니다. 명시적 에러 시그널링이 핵심입니다.

도구 정의의 토큰 비용도 중요합니다. 도구 정의 전체가 입력 토큰으로 계산되어, 도구가 많으면 매 요청 비용이 증가합니다. 이것이 사에서 오 개 규칙의 경제적 근거입니다. 다만 도구 정의는 정적 콘텐츠이므로 프롬프트 캐싱(Prompt Caching)으로 반복 비용을 줄일 수 있습니다.

이제 안티패턴 다섯 가지를 정리하겠습니다. 시험에서 이 안티패턴들을 식별하고 정답 패턴을 고르는 문제가 반복적으로 출제됩니다.

첫 번째 안티패턴은 스위스 아미 에이전트입니다. 단일 에이전트에 십오 개 이상의 도구를 몰아넣는 것입니다. 주의력 세금으로 도구 선택 정확도가 떨어지고, 최소 권한 원칙도 위반합니다. 정답 패턴은 사에서 오 개 도구를 가진 전문 서브에이전트로 분할하고 코디네이터 패턴을 적용하는 것입니다.

두 번째 안티패턴은 "더 나은 프롬프트로 해결"입니다. 십이 개 도구에서 오라우팅이 발생할 때, 도구 설명만 개선하려는 시도입니다. 도구 수 자체가 문제인데 설명만 고치면 근본 원인이 해결되지 않습니다. 도구 설명 개선은 사에서 오 개 이하일 때만 유효한 접근입니다. 십이 개 이상이면 반드시 서브에이전트 분할이 먼저입니다.

세 번째 안티패턴은 닷엠시피닷제이슨에 크레덴셜을 포함하는 것입니다. 버전 관리 시스템에 비밀값이 노출되어 보안 위반이 됩니다. 정답은 서버 정의는 프로젝트 파일에, 크레덴셜은 개인 설정 파일이나 환경변수에 분리하는 것입니다.

네 번째 안티패턴은 퓨샷(few-shot) 예시로 도구 호출 순서를 제어하려는 것입니다. 퓨샷은 출력 포맷을 맞추는 데에는 효과적이지만, 도구 호출 시퀀스를 제어하는 데에는 무효합니다. 정답 패턴은 프로그래밍적 전제조건(preconditions)을 사용하거나, 여러 단계를 단일 도구로 래핑하는 것입니다.

다섯 번째 안티패턴은 모호한 도구 설명입니다. "핸들즈 커스터머 스터프"처럼 뭉뚱그려 쓰는 것입니다. 클로드가 설명 기반으로 라우팅하므로 모호하면 오선택이 발생합니다. 정답은 긍정 경계와 부정 경계를 모두 포함한 구체적 설명에 입출력 명세를 더하는 것입니다.

시험 팁으로 마무리하겠습니다. 프롬프트 대 프로그래밍 강제의 구분을 확실히 하세요. 프롬프트는 확률적으로 실행되어 컨텍스트 압박 시 생략될 수 있지만, 포스트툴유즈 훅은 확정적으로 실행됩니다. 비즈니스 규칙, 특히 금액 한도나 컴플라이언스 관련 규칙은 반드시 훅으로 강제해야 합니다. "시스템 프롬프트를 더 강조해서 재작성한다"는 답은 항상 오답입니다.

셀프 리포티드 컨피던스(self-reported confidence) 함정도 기억하세요. 에이전트의 자체 보고 신뢰도는 보정되지 않은 값이므로, 이것을 금융 결정의 근거로 사용하면 항상 오답입니다.

마지막으로 외울 숫자를 정리합니다. 삼은 엠시피 프리미티브 수. 사에서 오는 에이전트당 최적 도구 수. 십팔 퍼센트는 이 도메인의 시험 가중치. 약 십일은 예상 출제 문항 수. 십오 이상은 슈퍼에이전트 안티패턴의 임계값. 일 대 엔 대 엔은 호스트 대 클라이언트 대 서버의 관계입니다. 이 숫자들을 확실히 기억해 두시면 시험에서 빠르게 정답을 고를 수 있습니다. 도메인 투 강의를 마치겠습니다.
