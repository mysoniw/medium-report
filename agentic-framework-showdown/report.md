# 아젠틱 코딩 프레임워크 대격돌: Superpowers vs. BMAD vs. SpecKit vs. GSD vs. OpenSpec

> 원문: [The Great Framework Showdown](https://medium.com/@richardhightower/the-great-framework-showdown-superpowers-vs-bmad-vs-speckit-vs-gsd-360983101c10) — Rick Hightower, 2026년 3월 (16분 분량)

---

## 1. 요약

5개 아젠틱 코딩 프레임워크가 합산 **170,000+ GitHub Stars**를 기록하며, "바이브 코딩(Vibe Coding)" 시대를 대체할 구조적 AI 개발 방법론으로 경쟁 중이다. 각 프레임워크는 **AI 에이전트에게 얼마나 많은 구조를 부여해야 하는가**라는 질문에 서로 다른 답을 제시한다.

| 프레임워크 | GitHub Stars | 핵심 포지셔닝 |
|-----------|-------------|-------------|
| **SpecKit** | 75,900 | 게이트 기반 스펙 프로세스 (GitHub 공식) |
| **BMAD** | 40,200 | 엔터프라이즈 팀 시뮬레이션 |
| **OpenSpec** | 29,500 | 브라운필드(기존 코드베이스) 특화 델타 스펙 |
| **GSD** | 28,100 | 컨텍스트 격리 + 웨이브 병렬 실행 |
| **Superpowers** | 성장 중 | TDD 강제 + 규율 시스템 |

---

## 2. 공통 구조: Agent-Workflow-Skill 삼각축

모든 프레임워크는 표면적 차이에도 불구하고 동일한 세 가지 기본 요소를 공유한다.

### Agent (에이전트)
시스템이 채택하는 페르소나 또는 역할. 제한된 전문 영역과 정의된 책임을 가진다.

- **BMAD**: 12+ 페르소나 (분석가, 아키텍트, 개발자, 스크럼 마스터 등)
- **GSD**: 서브에이전트 (리서처, 플래너, 실행자)
- **Superpowers**: 맥락 기반 자동 트리거 스킬(게이트)
- **SpecKit**: CLI 커맨드에 에이전트 동작 내장

### Workflow (워크플로우)
에이전트를 파이프라인으로 연결하는 시퀀스.

- **BMAD**: 4단계 애자일 사이클
- **SpecKit**: 게이트가 있는 순차 단계
- **GSD**: 의존성 기반 웨이브 실행
- **Superpowers**: 브레인스토밍→코드리뷰까지 자동 파이프라인

### Skill (스킬)
에이전트가 수행하는 원자적 단위 작업 (PRD 작성, 테스트 실행, 코드 리뷰 등).

> **핵심 인사이트**: 마케팅이 암시하는 것만큼 프레임워크 간 차이가 크지 않다. 모두 같은 구조적 문제(복잡한 소프트웨어 개발을 품질 게이트가 있는 관리 가능한 단계로 분해)를 풀고 있으며, 차이는 **어떤 단계를 강조하고, 게이트를 얼마나 엄격히 적용하느냐**에 있다.

## 3. 프레임워크별 심층 분석

### 3.1 BMAD — 엔터프라이즈 팀 시뮬레이터

**철학**: 12+ AI 페르소나로 전체 애자일 팀을 시뮬레이션

**작동 방식**:
1. **Analysis** → 1페이지 PRD 생성
2. **Planning** → 유저 스토리 + 인수 기준 분해
3. **Solutioning** → 아키텍트가 설계, 개발자가 구현 계획
4. **Implementation** → 스토리 단위 반복 구현

**차별점**:
- **문서 중심 철학**: 소스코드만이 진실의 원천이 아님. PRD, 아키텍처 스케치, 유저 스토리가 코드와 동등한 1급 산출물
- **Party Mode**: 단일 세션에서 여러 페르소나가 협업 (아키텍트가 개발자와 대화 중 디자인 리뷰 가능)
- **규모 적응형 지능**: 프로젝트 복잡도에 따라 문서화 수준 자동 조정

| 강점 | 약점 |
|------|------|
| 최고 수준의 추적성 (감사 추적, 컴플라이언스) | 가파른 학습 곡선 |
| 34+ 핵심 워크플로우, 1,468 커밋, 123 기여자 | 솔로 개발자에게 과도한 오버헤드 |
| 의사결정마다 문서화 연쇄 | 파일 기반 조율로 빠른 설계 변경 시 마찰 |

**적합한 경우**: 규제 산업(의료, 금융, 정부 계약), 다수 이해관계자, 감사 추적이 필수인 엔터프라이즈 팀

### 3.2 SpecKit — GitHub의 게이트 프로세스

**철학**: "스펙이 코드를 위해 존재하지 않는다. 코드가 스펙을 위해 존재한다."

**게이트 프로세스** (순차, 검증 통과 전 다음 단계 진입 불가):

```
Constitution → Specify → Plan → Tasks → Implement
  (선택: Clarify, Analyze)
```

**산출물**: spec.md, plan.md, research.md, data-model.md, 계약서, 퀵스타트 가이드

| 강점 | 약점 |
|------|------|
| AI 코딩의 가장 흔한 실패 방지 (요구사항 확정 전 구현 돌진) | 기능당 1~3시간+ 산출물 생성/검토 |
| 20+ AI 에이전트 지원 (가장 높은 플랫폼 호환성) | 정적 스펙 → 구현 중 자동 업데이트 없음 |
| 75,900 Stars (가장 높은 커뮤니티 규모) | 멀티에이전트 오케스트레이션 없음 |

**핵심 베팅**: 과잉 명세의 비용이 항상 과소 명세의 비용보다 낮다

---

### 3.3 OpenSpec — 브라운필드 특화

**철학**: "유연하되 경직되지 않게, 반복적이되 폭포수가 아니게, 간단하되 복잡하지 않게, 브라운필드를 위해 만들어짐"

**핵심 혁신 — 델타 스펙**:
- 기존 기능 수정 시 전체 스펙 재작성 대신 `ADDED`, `MODIFIED`, `REMOVED` 마커 사용
- 변경사항별 독립 폴더: `openspec/changes/<name>/`
- 변경당 약 250줄 출력 (SpecKit 약 800줄 대비)

| 비교 항목 | SpecKit | OpenSpec |
|----------|---------|---------|
| **최적 대상** | 그린필드 (새 프로젝트) | 브라운필드 (기존 코드베이스) |
| **의식(ceremony) 수준** | 높음 (게이트 강제) | 낮음 (자유 순서) |
| **산출물 크기** | ~800줄/변경 | ~250줄/변경 |
| **패스트포워드** | 없음 | `/opsx:ff` (전체 산출물 일괄 생성) |

**핵심 트레이드오프**: 스펙 깊이를 속도와 교환. 누적 변경의 전체 아키텍처 영향을 캡처하지 못할 위험

---

### 3.4 GSD (Get "Stuff" Done) — 컨텍스트 엔지니어링 파워하우스

**해결하는 문제 — 컨텍스트 부패(Context Rot)**:

AI 모델이 단일 세션에서 더 많은 토큰을 처리할수록 출력 품질이 저하되는 현상:

| 컨텍스트 사용률 | 품질 상태 |
|---------------|----------|
| 0~30% | 최고 품질 |
| 50%+ | 급하게 처리, 세부사항 건너뜀 |
| 70%+ | 환각 증가 |
| 80%+ | 이전 요구사항 망각 |

**해법 — 프레시 컨텍스트 + 웨이브 병렬 실행**:
- 각 실행 단위마다 **새로운 서브에이전트 컨텍스트** 생성
- 50번째 태스크도 1번째 태스크와 동일한 200K 토큰 깨끗한 창에서 시작
- 의존성 기반 **웨이브** 구조:
  - Wave 1: 독립 태스크 병렬 실행
  - Wave 2: Wave 1 결과 의존 태스크 병렬 실행
  - Wave 3: 이전 웨이브 전체 필요 태스크 순차 실행

**멀티 에이전트 구성**:
- 4개 병렬 리서처 (코드베이스 조사)
- 플래너 (연구→실행 계획 변환)
- 플랜 체커 (실행 전 계획 검증)
- 웨이브 기반 병렬 실행자
- 검증자 (목표-역방향 검증: "이것이 작동하려면 무엇이 TRUE여야 하는가?")
- 디버거 (과학적 가설 검증)

| 강점 | 약점 |
|------|------|
| 가장 실행 중심적 프레임워크 | 토큰 소비 높음 (50 태스크: 단일 세션 $5 → GSD $25~40) |
| 컨텍스트 격리로 일관된 품질 보장 | 계획 단계에서 협업적 스펙 정제 부족 |
| 태스크별 원자적 git 커밋 → 깨끗한 bisect/revert | 플랫폼 지원 좁음 (Claude Code, OpenCode, Gemini CLI) |
| Amazon, Google, Shopify 프로덕션 사용 | 단순 태스크에 과도할 수 있음 |

**저자 평가**: "하나만 골라야 한다면, GSD로 시작하라. 대부분의 카테고리에서 최고가 아니면 항상 2위."

---

### 3.5 Superpowers — 규율 강제 시스템

**철학**: 스펙 프레임워크도, 팀 시뮬레이터도, 컨텍스트 엔진도 아닌 **규율 강제 시스템**

**차별화 요소**:

1. **브레인스토밍이 하드 게이트**: 설계가 승인되기 전까지 코드 작성, 프로젝트 스캐폴딩, 어떤 구현 행위도 **금지**. 우회 불가, "이건 너무 단순해서 설계 불필요" 탈출구 없음

2. **TDD가 철칙**: 실패하는 테스트 없이 프로덕션 코드를 작성하면 → **삭제**. 참고용 저장도 없고, 적응도 없고, 그냥 삭제

3. **설득 기반 가드레일**: AI가 단계를 건너뛰기 위해 사용하는 합리화를 명시적으로 이름 붙이고 사전 차단
   - "이건 테스트하기엔 너무 단순해"
   - "나중에 테스트 쓸게"
   - "빨리 이것만 고칠게"

4. **2단계 리뷰**: 서브에이전트 출력마다 스펙 준수 리뷰 + 코드 품질 리뷰 통과 필수

| 강점 | 약점 |
|------|------|
| 가장 일관되게 높은 품질의 출력 | 속도 희생 (원래 10분 → 30분) |
| 가장 엄격한 테스트 강제 | 경험 많은 개발자가 잘 이해하는 태스크에서 답답할 수 있음 |
| AI 심리학 대응 (합리화 차단) | 프로토타입/일회성 코드에 부적합 |

## 4. 의사결정 가이드

### 상황별 프레임워크 선택

| 상황 | 추천 프레임워크 | 이유 |
|------|--------------|------|
| 규제 산업 엔터프라이즈 | **BMAD** | 감사 추적, 컴플라이언스 문서화 자연 생성 |
| 새 프로젝트 (그린필드) | **SpecKit** | 가장 엄격한 스펙 프로세스, 조기 구현 방지 |
| 기존 코드베이스 유지보수 | **OpenSpec** | 델타 스펙으로 경량 반복 변경 |
| 복잡한 기능, 병렬 컴포넌트 | **GSD** | 컨텍스트 격리, 웨이브 병렬, 일관된 품질 |
| 코드 품질 최우선 | **Superpowers** | 강제 TDD, 2단계 리뷰 |
| **하나만 골라야 한다면** | **GSD** | 대부분 카테고리에서 1위 또는 2위 |

### 하이브리드 조합 전략

| 조합 | 용도 |
|------|------|
| **SpecKit + GSD** | 최강 스펙 계층 + 최강 실행 계층 (도구 복잡성 2배) |
| **BMAD + Superpowers** | BMAD 추적성 + Superpowers 코드 품질 (엔터프라이즈) |
| **OpenSpec + Superpowers** | 최소 의식 + 테스트 규율 강제 (브라운필드) |

## 5. 핵심 인사이트 정리

1. **바이브 코딩 시대는 끝났다**: AI 보조 개발에 구조를 추가할지 여부가 아니라, **얼마나, 어떤 종류의 구조**를 추가할지가 질문이다

2. **모든 프레임워크는 같은 삼각축**(Agent, Workflow, Skill)을 공유한다 → 조합(하이브리드) 가능

3. **컨텍스트 부패는 실제 문제**다: 긴 세션에서 AI 출력 품질은 측정 가능하게 저하된다. GSD와 Superpowers가 이를 프레시 컨텍스트로 해결

4. **트레이드오프는 불가피**하다:
   - 포괄성 vs. 민첩성 (BMAD)
   - 스펙 깊이 vs. 속도 (SpecKit ↔ OpenSpec)
   - 토큰 비용 vs. 출력 품질 (GSD)
   - 신뢰성 vs. 속도 (Superpowers)

5. **저자의 최종 선호**: GSD와 Superpowers에 기울어짐. 정확성과 검증의 비용은 그만한 가치가 있다

## 7. 원문 영어 표현 해설

원문 저자가 사용한 표현 중, 단순 번역으로는 놓치기 쉬운 뉘앙스를 가진 단어와 구문을 정리합니다.

### 핵심 개념어

| 원문 표현 | 직역 | 저자가 의도한 뉘앙스 |
|----------|------|-------------------|
| **Vibe Coding** | 분위기 코딩 | AI에게 대충 지시하고 "감"으로 결과를 받아들이는 비구조적 코딩 방식. 부정적 뉘앙스 — "대충 느낌으로 때려넣는 코딩" |
| **Viable Coding** | 실행 가능한 코딩 | Vibe와 발음이 비슷한 말장난(Vibe → Viable). "감으로 하는 코딩"에서 "실제로 쓸 수 있는 코딩"으로의 전환 |
| **Context Rot** | 컨텍스트 부패 | 음식이 썩듯이(rot) AI의 대화 맥락이 길어지면 품질이 **부패**해간다는 비유. 기술 용어가 아니라 저자가 만든 은유 |
| **Hallucination** | 환각 | AI가 사실이 아닌 것을 자신 있게 말하는 현상. 사람이 환각을 보는 것에 빗댄 AI 업계 용어 |
| **Brownfield** | 갈색 땅 | 이미 건물이 있던 부지(오염된 재개발지). 소프트웨어에서는 **기존 코드가 이미 있는** 프로젝트. "지저분하지만 현실" |
| **Greenfield** | 녹색 땅 | 아무것도 없는 깨끗한 초원. 소프트웨어에서는 **처음부터 새로 시작**하는 프로젝트. "깨끗하지만 비현실적일 수 있음" |

### 비유·수사 표현

| 원문 표현 | 문맥 | 뉘앙스 |
|----------|------|--------|
| **"full machine shop"** | BMAD를 설명하며 "다른 것들이 전동공구면, BMAD는 완전한 기계 공장" | 단순한 도구가 아니라 **공장 전체**라는 스케일 강조. 강력하지만 과할 수 있다는 이중 의미 |
| **"sledgehammer on a thumbtack"** | SpecKit의 과도한 의식(ceremony)을 비판하며 | "압정 박는 데 대형 망치를 쓴다" — 한국어의 "닭 잡는 데 소 잡는 칼" 과 같은 표현 |
| **"Choosing Your Weapon"** | 프레임워크 선택 가이드 제목 | RPG 게임에서 무기를 고르는 느낌. "도구"가 아니라 "무기"라고 한 건 **전장(실무)에서 생존이 걸린 선택**이라는 뉘앙스 |
| **"iron law"** | Superpowers의 TDD 정책 | "철의 법칙" — 예외 없는 절대 규칙. 부드러운 가이드라인이 아니라 **깨면 안 되는 불문율** |
| **"the debate is over"** | 구조적 AI 워크플로우 필요성에 대해 | "논쟁은 끝났다" — 더 이상 찬반을 따질 단계가 아니라는 단호한 선언. 업계 컨센서스가 형성됐다는 자신감 |
| **"ships code"** | GSD가 코드를 출하한다 | "ship"은 소프트웨어를 **세상에 내보내다**. "write"가 아니라 "ship"을 쓴 건 완성·배포까지 간다는 뜻 |
| **"paper trail"** | BMAD의 추적성 장점 | 서류 흔적 — 범죄 수사처럼 **의사결정의 흔적을 남긴다**. 감사(audit) 문화에서 온 표현 |

### 업계 전문 용어

| 원문 표현 | 의미 | 왜 이 단어를 쓰는가 |
|----------|------|-------------------|
| **Ceremony** | 의식, 절차 | 소프트웨어에서 "ceremony"는 실제 작업 전에 거쳐야 하는 **형식적 절차**를 뜻함. 부정적 뉘앙스 — 관료주의적 번거로움 |
| **Gate / Gated** | 관문 | 공항 탑승구(gate)처럼 **통과 조건을 충족해야 다음으로 넘어갈 수 있는** 체크포인트 |
| **Guardrails** | 가드레일 | 고속도로 난간처럼 AI가 **벗어나면 안 되는 경계**를 설정하는 안전장치 |
| **Traceability** | 추적성 | 식품 원산지 추적처럼, 코드의 **모든 결정이 어디서 왜 나왔는지** 역추적 가능한 성질 |
| **Opinionated** | 주관이 강한 | "이렇게 해야 한다"고 **강하게 방향을 정해놓은** 프레임워크. 유연함의 반대. 칭찬이 될 수도, 비판이 될 수도 있음 |
| **Trade-off** | 등가교환 | 하나를 얻으면 하나를 잃는 관계. 저자가 매 프레임워크마다 이 단어를 반복한 건 **공짜 점심은 없다**는 메시지 |
| **Primitives** | 원시 요소 | 프로그래밍의 기본 자료형처럼, 더 이상 쪼갤 수 없는 **가장 기본적인 구성 단위** |
| **Persona** | 페르소나 | 원래 연극의 가면. BMAD에서 AI가 **특정 역할의 가면을 쓰고** 그 전문가처럼 행동한다는 의미 |
| **Vertical Slice** | 수직 절단 | 케이크를 세로로 자르면 모든 층이 보이듯, 프론트부터 백엔드까지 **한 기능의 전체 계층을 관통**하는 작업 단위 |
| **Social Engineering** | 사회공학 | 원래 해킹 용어(사람을 속여 정보를 빼내는 기법). 여기서는 **AI를 말로 꼬드겨 규칙을 어기게 만드는** 시도 |
| **Delta** | 변화량 | 수학의 Δ(변화). 전체를 다시 쓰지 않고 **바뀐 부분만** 기술하는 방식 |
| **Orchestration** | 오케스트레이션 | 오케스트라 지휘처럼, 여러 에이전트를 **조율하고 동기화**하는 것 |
| **Git Bisect** | 이진 탐색 디버깅 | git 기능. 버그가 처음 발생한 커밋을 **반씩 나눠가며** 찾는 기법. "atomic commit"이 이걸 가능하게 한다는 맥락 |

### 저자의 어투에서 느껴지는 것

저자 Rick Hightower는 의도적으로 **전투·게임·공학 은유**를 많이 사용합니다:
