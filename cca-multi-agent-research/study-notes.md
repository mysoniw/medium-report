# CCA 멀티에이전트 리서치 시나리오 — 스터디 노트

## 1. 영한 용어표 (English-Korean Glossary)

| English Term | 한국어 | 설명 | 빈도 |
|-------------|--------|------|------|
| hub-and-spoke | 허브앤스포크 | 중앙 코디네이터가 모든 라우팅을 담당하는 분산 아키텍처 패턴 | ★★★ |
| coordinator | 코디네이터 | 허브 역할. 서브에이전트에 작업 위임 및 결과 종합 | ★★★ |
| sub-agent | 서브에이전트 | 스포크 역할. 전문화된 작업을 수행하는 개별 에이전트 | ★★★ |
| context isolation | 컨텍스트 격리 | 서브에이전트가 코디네이터 컨텍스트를 자동 상속하지 않는 원칙 | ★★★ |
| explicit context passing | 명시적 컨텍스트 전달 | 서브에이전트에 필요한 정보를 명시적으로 포함하여 위임 | ★★★ |
| blank slate | 빈 상태 | 서브에이전트의 초기 상태 — 아무 컨텍스트도 없음 | ★★ |
| context forking | 컨텍스트 포킹 | Unix fork()처럼 필요한 컨텍스트만 선택적으로 복제 | ★★ |
| super-agent anti-pattern | 슈퍼에이전트 안티패턴 | 단일 에이전트에 과도한 도구를 부여하는 잘못된 설계 | ★★★ |
| attention tax | 주의력 세금 | 도구가 많을수록 매 선택에서 지불하는 인지적 비용 | ★★ |
| silent failure | 사일런트 실패 | 에러가 성공으로 위장되어 감지되지 않는 실패 모드 | ★★★ |
| structured error context | 구조화된 에러 컨텍스트 | 에러 유형, 소스, 재시도 가능 여부 등을 포함한 에러 응답 | ★★ |
| negative bounds | 부정 경계 | 도구 설명에서 "하지 않는 것"을 명시하는 것 | ★★ |
| MCP primitives | MCP 프리미티브 | Tools(동사), Resources(명사), Prompts(패턴)의 3대 구성요소 | ★★ |
| task decomposition DAG | 작업 분해 방향 비순환 그래프 | 빌드 시스템 의존성 그래프 원리로 병렬/순차 실행 결정 | ★★ |
| validation-retry loop | 유효성 검사-재시도 루프 | 출력 검증 후 실패 시 에러 피드백과 함께 재시도 | ★ |
| conflict resolution | 충돌 해결 | 소스 간 상충 시 신뢰도 랭킹, 다수결, 인간 에스컬레이션 | ★★ |
| source reliability ranking | 소스 신뢰도 랭킹 | peer-reviewed > 공식 문서 > 뉴스 > 블로그 > 소셜 미디어 | ★ |
| misrouting | 잘못된 라우팅 | 부정 경계 누락으로 도구를 잘못된 용도로 호출 | ★ |

> ★★★ = 시험에 반복 출제 / ★★ = 핵심 개념 / ★ = 보조 개념

## 2. 시험 함정 Top 5 (Exam Trap Top 5)

### 함정 1: "서브에이전트는 코디네이터 컨텍스트를 자동으로 상속한다"

**왜 함정인가**: 멀티스레드 프로그래밍의 공유 메모리 멘탈 모델에서 오는 오해. 부모가 아는 것을 자식도 안다고 착각.

**정답**: 서브에이전트는 **blank slate**(빈 상태)에서 시작. 코디네이터가 **명시적으로 전달**(explicit context passing)한 것만 안다. 전달하지 않은 지시 → 서브에이전트는 모른다.

**킬러 문제**: "코디네이터가 APA 인용 형식을 요구했는데 서브에이전트가 MLA를 반환했다. 원인은?" → **인용 형식 요구사항이 서브에이전트에 전달되지 않았다**

---

### 함정 2: "도구를 많이 부여할수록 에이전트 성능이 향상된다"

**왜 함정인가**: 직관적으로 역량이 많으면 좋다고 생각. "더 많은 도구 = 더 유능한 에이전트"라는 유혹.

**정답**: 에이전트당 **4-5개 도구**가 최적. 18개 도구 = **주의력 세금**(attention tax)으로 인한 성능 저하. SDK 하드 리밋이 아닌 **아키텍처 모범 사례**.

**구분**: 이 제한은 기술적 제약이 아니라 설계 원칙. "SDK가 도구 수를 제한한다"는 오답.

---

### 함정 3: "서브에이전트 간 직접 통신이 효율적이다"

**왜 함정인가**: 중간 단계를 건너뛰면 더 빠를 것 같은 직관.

**정답**: 허브앤스포크에서 **스포크끼리는 절대 직접 통신하지 않는다**. 모든 조율은 허브(코디네이터)를 통해서만. "서브에이전트 A가 서브에이전트 B에 직접 결과를 전달"은 항상 오답.

**뉴스룸 비유로 기억**: 기자끼리 기사를 공유하지 않는다. 편집자가 중개한다.

---

### 함정 4: "서브에이전트가 null을 반환하면 데이터가 없는 것이다"

**왜 함정인가**: `{"status": "success", "data": null}`을 "관련 데이터 없음"으로 해석하는 실수.

**정답**: **"데이터 없음"과 "소스 접근 불가"는 완전히 다른 상황**. 구조화된 에러 컨텍스트(structured error context)로 에러 유형, 소스, 재시도 가능 여부를 구분해야 한다.

**사일런트 실패의 위험**: 반박 증거가 누락된 채 "완전한" 보고서가 생성 → 잘못된 결론

---

### 함정 5: "먼저 응답한 소스가 더 신뢰할 수 있다"

**왜 함정인가**: 응답 속도와 신뢰도를 혼동.

**정답**: 충돌 해결에는 체계적 전략이 필요. **소스 신뢰도 랭킹**(peer-reviewed > 공식 문서 > 뉴스 > 블로그 > 소셜 미디어), **다수결 합의**, 해결 불가능한 고위험 주장은 **인간 에스컬레이션**. "First result wins"는 안티패턴.

## 3. 연습 문제 5문항 (Practice Questions)

### Q1. 컨텍스트 격리

멀티에이전트 리서치 시스템에서 코디네이터가 문서 분석기(Document Analyzer)에 "경쟁사 제품 비교 분석"을 위임했다. 코디네이터의 컨텍스트에는 "분석 대상은 2025년 이후 출시 제품만"이라는 조건이 있다. 문서 분석기가 2020년 제품까지 포함한 결과를 반환했다. 근본 원인은?

- A) 문서 분석기의 도구가 날짜 필터링을 지원하지 않았다
- B) 코디네이터가 날짜 조건을 문서 분석기에 명시적으로 전달하지 않았다
- C) 문서 분석기가 코디네이터 컨텍스트 상속에 실패했다
- D) MCP 프로토콜이 날짜 필터를 지원하지 않는다

**정답: B**
> 서브에이전트는 코디네이터 컨텍스트를 상속하지 않는다(context isolation). 명시적으로 전달(explicit context passing)한 것만 안다. C가 함정 — "상속 실패"라는 표현은 상속이 기본 동작이라는 전제를 깔고 있어 오답.

---

### Q2. 슈퍼에이전트 안티패턴

다음 중 멀티에이전트 리서치 시스템의 올바른 설계 원칙은?

- A) 단일 에이전트에 모든 도구를 부여하여 컨텍스트 전환 비용을 제거한다
- B) 각 에이전트에 4-5개의 집중된 도구를 부여하고, 코디네이터가 오케스트레이션한다
- C) 도구 수에 SDK 하드 리밋이 있으므로 에이전트를 분리해야 한다
- D) 서브에이전트가 필요에 따라 다른 서브에이전트의 도구를 호출할 수 있도록 한다

**정답: B**
> 4-5개 도구/에이전트가 모범 사례. A는 슈퍼에이전트 안티패턴(attention tax). C는 SDK 제한이 아니라 아키텍처 원칙. D는 허브앤스포크 위반(스포크 간 직접 통신 불가).

### Q3. 사일런트 실패

팩트 체커(Fact Checker) 서브에이전트가 외부 API 호출 시 타임아웃이 발생했다. 다음 중 올바른 응답 형식은?

- A) `{"status": "success", "data": null}`
- B) `{"status": "error", "error_type": "timeout", "source": "api.example.com", "retry_eligible": true}`
- C) 빈 응답을 반환하고 코디네이터가 타임아웃을 추론하도록 한다
- D) 에러를 무시하고 가용한 부분 데이터만 반환한다

**정답: B**
> 구조화된 에러 컨텍스트(structured error context)가 정답. A는 사일런트 실패의 전형 — 에러를 성공으로 위장. C는 코디네이터에 불필요한 추론 부담. D는 데이터 갭 정보 누락.

---

### Q4. MCP 프리미티브

MCP(Model Context Protocol)의 세 가지 프리미티브와 그 유추가 올바르게 짝지어진 것은?

- A) Tools = 명사, Resources = 동사, Prompts = 패턴
- B) Tools = 동사, Resources = 패턴, Prompts = 명사
- C) Tools = 동사, Resources = 명사, Prompts = 패턴
- D) Tools = 패턴, Resources = 명사, Prompts = 동사

**정답: C**
> Tools는 실행 가능한 함수(동사), Resources는 데이터 스키마/카탈로그(명사), Prompts는 공통 작업용 템플릿(패턴). "동사, 명사, 패턴"으로 외운다.

### Q5. 작업 분해 및 충돌 해결

멀티에이전트 리서치 시스템에서 웹 리서처와 문서 분석기가 같은 주제에 대해 상충하는 결과를 반환했다. 웹 리서처의 소스는 기술 블로그 3곳, 문서 분석기의 소스는 동료 심사 논문 1편이다. 올바른 충돌 해결 접근법은?

- A) 먼저 응답한 에이전트의 결과를 채택한다
- B) 소스 수가 더 많은 웹 리서처의 결과를 채택한다
- C) 소스 신뢰도 랭킹에 따라 동료 심사 논문의 우선순위를 높인다
- D) 두 결과를 모두 폐기하고 새로운 검색을 수행한다

**정답: C**
> 소스 신뢰도 랭킹: peer-reviewed > 공식 문서 > 뉴스 > 블로그 > 소셜 미디어. 수량이 아닌 품질이 기준. A는 "first result wins" 안티패턴. B는 소스 수에 의존하는 오류. D는 불필요한 비용.

## 5. 핵심 개념 요약 (Key Concepts)

### Coordinator-Subagent (코디네이터-서브에이전트)

```
코디네이터(Hub)
├── 작업 위임 (delegate_task)
├── 결과 수집 (collect_results)
├── 충돌 해결 (resolve_conflicts)
└── 보고서 작성 (compile_report)

서브에이전트(Spoke) × 4
├── 전문 도구 4-5개
├── blank slate에서 시작
├── 명시적으로 전달받은 컨텍스트만 보유
└── 코디네이터에만 결과 반환 (다른 스포크와 직접 통신 불가)
```

### Hub-and-Spoke (허브앤스포크)

- **출처**: 항공 네트워크 설계 (인천공항 = 허브, 각 도시 노선 = 스포크)
- **핵심 제약**: 스포크 간 직접 통신 없음. 모든 데이터 흐름은 허브 경유.
- **뉴스룸 비유**: 편집자 = 허브, 기자들 = 스포크. 기자끼리 기사 공유 안 함.
- **시험 출제 포인트**: "서브에이전트 A가 B에 직접 전달" → 항상 오답

### Context Isolation (컨텍스트 격리)

- **원칙**: 서브에이전트는 코디네이터 컨텍스트를 **상속하지 않는다**
- **초기 상태**: blank slate (빈 상태)
- **해결책**: explicit context passing (명시적 컨텍스트 전달)
- **포킹**: 필요한 컨텍스트만 선택적으로 복제 (Unix fork() 유사)
- **전달 항목**: 작업 설명, 제약 조건, 출력 형식, 필요 컨텍스트
- **비전달 항목**: 전체 대화 이력, 타 에이전트 결과, 전체 계획 (명시적 전달 전까지)

### 시험 치트시트 (Exam Cheat Sheet)

