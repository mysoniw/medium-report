# CCA 모의시험: 60문항 상세 해설

CCA 시험 준비 시리즈의 최종편(8/8)입니다. 실제 Claude Certified Architect Foundations 시험을 미러링하는 60문항 모의시험으로, 5개 역량 도메인 전체를 공식 가중치에 맞게 배분했습니다. 각 문항에 **오답이 왜 틀린지**까지 상세히 설명하여 단순 암기가 아닌 개념 이해를 유도합니다.

**시험 파라미터:**
- **문항 수:** 60문항
- **시간:** 120분
- **합격 기준:** 720/1000 (~43/60 정답, 72%)
- **권장 목표:** 50+/60 (83%+)
- **난이도 분포:** Easy 18 / Medium 27 / Hard 15

## 도메인 1: 에이전틱 아키텍처 & 오케스트레이션 (Q1–Q16)

### 문제 1 (Easy)
한 회사가 중앙 에이전트가 전문 에이전트들(웹 검색, 문서 분석, 팩트 체크)에게 작업을 할당하고 결과를 통합 보고서로 조합하는 리서치 시스템을 구축해야 합니다. 어떤 멀티에이전트 패턴이 가장 적합합니까?

A) 모든 에이전트가 직접 통신하는 피어투피어 메시(peer-to-peer mesh)
B) **허브앤스포크(hub-and-spoke) 토폴로지의 코디네이터-서브에이전트 패턴(coordinator-subagent pattern)**
C) 각 에이전트가 출력을 다음에 전달하는 순차 파이프라인(sequential pipeline)
D) 모든 에이전트가 공통 컨텍스트 저장소를 읽고 쓰는 공유 메모리 아키텍처(shared-memory architecture)

**정답: B**

**해설:** **코디네이터-서브에이전트 패턴**(허브앤스포크)은 CCA 시험에서 가장 많이 테스트되는 멀티에이전트 아키텍처입니다. 중앙 코디네이터가 전문 서브에이전트에 작업을 위임하고, 결과를 수집·합성합니다. A(피어투피어 메시)는 관리 불가능한 복잡성을 만들고 — 에이전트들은 서로 직접 조율하지 않으며 모든 조율은 코디네이터를 통해 이루어집니다. C(순차 파이프라인)는 **독립적이고 병렬적**인 작업에 적합하지 않습니다. D(공유 메모리)는 **컨텍스트 격리(context isolation)** 원칙을 위반합니다.

> **시험 함정:** 피어투피어 메시는 세련되어 보이지만 허브앤스포크 원칙을 위반합니다. CCA에서 코디네이터는 항상 단일 오케스트레이션 지점입니다.

### 문제 2 (Medium)
Claude가 대화 중 도구를 사용할 때 **에이전틱 루프(agentic loop)**의 올바른 단계 순서는?

A) generate → end_turn → tool_use → execute → result
B) tool_use → generate → execute → result → end_turn
C) **generate → tool_use → execute → result → generate → end_turn**
D) execute → tool_use → generate → result → end_turn

**정답: C**

**해설:** **에이전틱 루프**는 정확한 순서를 따릅니다: 모델이 응답을 **생성(generate)**하고, 여기에 **tool_use** 요청이 포함되며(`stop_reason: "tool_use"`로 표시), 도구가 외부에서 **실행(execute)**되고, **결과(result)**가 모델에 반환되며, 모델이 결과를 반영해 다시 **생성**하고, 마지막으로 **end_turn**을 신호합니다. A는 tool 실행 전에 end_turn을 잘못 배치합니다. B는 생성 전에 tool_use로 시작합니다. D는 모델이 도구 호출을 요청하기 전에 실행으로 시작합니다.

> **CCA 시험 팁:** `stop_reason: "tool_use"`는 모델이 도구 실행을 **요청**하는 것이지 실행이 완료된 것이 아닙니다. 루프를 계속하려면 도구 결과를 반환해야 합니다.

### 💡 문제 3 (Hard)
코디네이터 에이전트가 복잡한 리서치 작업을 서브에이전트에 위임합니다. 서브에이전트가 컨텍스트 인식 결과를 제공하려면 고객의 이전 대화 히스토리에 접근해야 합니다. 어떻게 구현해야 합니까?

A) 서브에이전트가 코디네이터의 전체 대화 히스토리를 자동으로 상속한다
B) 서브에이전트 생성 시 `shared_context=True` 파라미터를 사용한다
C) **서브에이전트의 프롬프트 문자열(prompt string)에 관련 컨텍스트를 명시적으로 포함한다**
D) 두 에이전트가 접근하는 공유 데이터베이스에 대화 히스토리를 저장한다

**정답: C**

**해설:** **서브에이전트는 코디네이터의 컨텍스트를 상속하지 않습니다.** 이것이 CCA 시험에서 가장 자주 테스트되는 개념입니다. Agent 도구로 서브에이전트를 생성할 때, 서브에이전트는 자신의 **프롬프트 문자열**과 **시스템 프롬프트**만 받습니다 — 그 외에는 아무것도 없습니다. 코디네이터가 보내는 프롬프트에 관련 정보를 **명시적으로 포함**해야 합니다. A는 가장 흔한 오해입니다. B는 존재하지 않는 파라미터입니다. D는 공유 상태를 도입하여 컨텍스트 격리를 위반합니다.

> **시험 함정:** "많은 사람이 서브에이전트가 컨텍스트를 상속한다고 가정합니다. 그렇지 않습니다." 이 한 문장이 CCA 시험의 가장 중요한 아키텍처 개념을 담고 있습니다.

### 문제 4 (Medium)
개발팀이 세 개의 독립적인 코드 리뷰 작업과, 세 리뷰가 모두 완료된 후에 의존하는 순차적 문서화 작업 하나를 처리해야 합니다. 최적의 아키텍처는?

A) 단일 에이전트가 네 작업을 모두 순차적으로 처리한다
B) 작업당 하나씩 네 개의 별도 에이전트가 병렬로 실행한다
C) **독립 리뷰를 위한 세 개의 병렬 서브에이전트 + 코디네이터가 결과를 종합하여 문서화 작업 수행**
D) 각 리뷰가 다음으로 전달되는 파이프라인 아키텍처

**정답: C**

**해설:** 올바른 분해 전략: **독립 작업 → 병렬 서브에이전트; 의존 작업 → 코디네이터 합성**. 세 코드 리뷰는 독립적이므로 서브에이전트로 병렬 실행 가능합니다. 문서화 작업은 세 결과 모두를 필요로 하므로 코디네이터가 서브에이전트 출력을 수집한 후 처리해야 합니다.

> **CCA 시험 팁:** 분해 의사결정 트리: 순차적+공유 컨텍스트 → 단일 에이전트. 독립적+병렬 → 서브에이전트. 이전 결과에 의존 → 코디네이터 합성.

### 문제 5 (Medium)
멀티에이전트 리서치 시스템의 서브에이전트가 웹 페이지를 가져오는 중 네트워크 에러를 만납니다. 올바른 에러 처리 방식은?

A) 빈 결과를 반환하여 코디네이터가 다른 서브에이전트와 계속 진행하도록 한다
B) 요청이 성공할 때까지 무한 재시도한다
C) 처리되지 않은 예외를 던져 전체 시스템을 종료한다
D) **`{"status": "failure", "error": "network_timeout", "details": "..."}`와 같은 구조화된 상태 객체를 반환한다**

**정답: D**

**해설:** **사일런트 실패(silent failure)**는 멀티에이전트 시스템의 치명적 안티패턴입니다. 올바른 접근은 성공/실패/부분 완료를 명확히 나타내는 **구조화된 상태 객체**를 반환하는 것입니다. A는 사일런트 실패 안티패턴 — `{"status": "success", "data": null}`은 코디네이터가 "데이터 없음"으로 오해하게 만듭니다. B(무한 재시도)는 시스템 전체를 멈출 수 있습니다. C(미처리 예외)는 과도하게 파괴적입니다.

### 문제 6 (Medium)
Claude API에서 어떤 `tool_choice` 설정이 Claude가 특정 이름의 도구를 호출하도록 **보장**합니까?

A) `tool_choice: "auto"` — Claude가 가장 적절한 도구를 선택한다
B) `tool_choice: "any"` — Claude가 최소 하나의 도구를 사용해야 한다
C) **`tool_choice: {"type": "tool", "name": "extract_data"}` — 특정 이름의 도구를 강제한다**
D) `tool_choice: "none"` — 모든 도구를 비활성화하고 텍스트 출력을 강제한다

**정답: C**

**해설:** **named-tool** 형식(`{"type": "tool", "name": "X"}`)만이 Claude가 특정 도구를 호출하도록 **보장**합니다. `auto`는 모델이 결정하며 도구를 사용하지 않을 수도 있습니다. `any`는 도구 사용을 강제하지만 어떤 도구인지 지정하지 않습니다. `none`은 도구를 완전히 비활성화합니다.

> **CCA 시험 팁:** 네 가지 `tool_choice` 값을 완벽히 외우세요: `auto`(모델 결정), `any`(도구 사용 강제), `none`(도구 금지), `{"type": "tool", "name": "X"}`(특정 도구 강제).

### 💡 문제 7 (Hard)
고객 지원 시스템이 복잡한 이슈를 탐색하기 위한 대화의 전문 분기를 생성해야 하며, 메인 대화 흐름으로 돌아갈 수 있는 능력을 보존해야 합니다. 어떤 기능이 이를 가능하게 합니까?

A) 이전 컨텍스트의 요약으로 새 대화를 생성한다
B) 전체 대화 히스토리를 프롬프트에 붙여넣어 서브에이전트를 사용한다
C) **`fork_session=True`를 사용하여 전체 대화 히스토리가 포함된 세션 분기를 생성한다**
D) 대화 상태를 데이터베이스에 저장하고 나중에 복원한다

**정답: C**

**해설:** **컨텍스트 포킹(context forking)**은 `fork_session=True`를 통해 현재 세션의 전체 대화 히스토리를 독립 분기로 완전 복사합니다. 메인 대화 흐름을 오염시키지 않고 복잡한 경로를 탐색할 수 있습니다.

### 문제 8 (Easy)
Claude API 호출에서 **시스템 프롬프트(system prompt)**의 주요 목적은 무엇입니까?

A) 각 요청에 대한 사용자별 선호도를 제공한다
B) **대화 전체에 적용되는 지속적인 행동 지침, 역할 정의, 제약 조건을 설정한다**
C) 모델의 기본 안전 가이드라인을 재정의한다
D) 세션 간 대화 히스토리를 저장한다

**정답: B**

**해설:** **시스템 프롬프트**는 에이전트의 지속적 컨텍스트 — 역할, 행동 규칙, 출력 형식 요구사항, 도메인 제약을 정의합니다. 매 대화 시작 시 로드되어 모든 후속 응답에 영향을 미칩니다.

### 💡 문제 9 (Hard)
고객 지원 에이전트가 복잡한 청구 분쟁을 해결하려 했으나 모든 가용 도구를 확인한 후에도 만족스러운 해결책을 찾지 못했습니다. 올바른 에스컬레이션 패턴은?

A) 에이전트가 신뢰도 점수(confidence score)를 보고하고 임계값 이하이면 에스컬레이션한다
B) 에이전트가 이슈를 해결할 때까지 점점 더 상세한 프롬프트로 재시도한다
C) **바운디드 리트라이(bounded retry, 2~3회 + 구체적 에러 피드백) → 전체 대화 히스토리와 함께 인간에게 에스컬레이션**
D) 고객 대기 시간을 최소화하기 위해 재시도 없이 즉시 에스컬레이션한다

**정답: C**

**해설:** 올바른 에스컬레이션 패턴은 **바운디드 리트라이**(최대 2~3회)로, 각 재시도에 이전 시도의 **구체적 에러 피드백**을 포함하고, 해결 실패 시 **전체 대화 히스토리와 함께 인간에게 에스컬레이션**합니다. A는 **보정되지 않은(not calibrated)** 자기보고 신뢰도를 사용합니다. B는 재시도 상한이 없어 무한 루프 가능성이 있습니다. D는 해결 시도 전에 리소스를 낭비합니다.

> **시험 함정:** 자기보고 신뢰도 점수(85%, 92% 등)는 에스컬레이션 문제에서 가장 흔한 오답입니다.

### 문제 10 (Easy)
단일 Claude 에이전트에 리서치 작업을 위한 18개의 도구가 할당되었습니다. 도구 선택 정확도가 크게 저하되었습니다. 권장 해결책은?

A) 도구 선택에 대한 더 명확한 안내를 제공하도록 시스템 프롬프트를 개선한다
B) 각 시나리오에 맞는 올바른 도구를 보여주는 few-shot 예시를 추가한다
C) **도구를 전문 서브에이전트로 분배하여 각각 4~5개 도구를 가지게 한다**
D) 도구 추론에 더 많은 공간을 주기 위해 컨텍스트 윈도우를 늘린다

**정답: C**

**해설:** **4~5개 도구 규칙**은 핵심 CCA 원칙입니다. 사용 가능한 도구 수가 증가하면 도구 선택 정확도가 측정 가능하게 저하됩니다. 올바른 패턴은 각각 4~5개의 집중된 도구 세트를 가진 **전문 서브에이전트**를 만드는 것입니다.

> **CCA 시험 팁:** 에이전트가 4~5개 이상의 도구로 문제를 겪는다는 설명이 나오면, 답은 전문 서브에이전트로의 분할입니다.

### 문제 11 (Easy)
코디네이터-서브에이전트 아키텍처에서 서브에이전트들은 서로 어떻게 통신합니까?

A) 서브에이전트들이 직접 에이전트 간 통신을 위한 메시지 버스를 공유한다
B) 서브에이전트들이 데이터를 전달하기 위해 공유 메모리 저장소를 사용한다
C) **서브에이전트들은 직접 통신하지 않는다 — 모든 조율은 코디네이터를 통해 이루어진다**
D) 서브에이전트들이 중간 결과를 교환하기 위해 피어투피어 프로토콜을 사용한다

**정답: C**

### 💡 문제 33 (Hard)
MCP(Model Context Protocol)의 세 가지 프리미티브와 각각의 사용 시점은?

A) Client, Server, Endpoint
B) Request, Response, Stream
C) **Tool(액션 실행), Resource(읽기 전용 데이터), Prompt(재사용 가능한 템플릿)**
D) Input, Process, Output

**정답: C**

**해설:** 세 가지 **MCP 프리미티브**: **Tool** — 무언가를 발생시키는 실행 가능한 함수(DB 쿼리, API 호출, 파일 쓰기); **Resource** — 컨텍스트를 위한 읽기 전용 데이터(문서, 스키마, 지식베이스); **Prompt** — 재사용 가능한 지침 템플릿/워크플로우. 핵심 결정: "Claude가 무언가를 **실행**해야 하는가?" → Tool. "Claude가 컨텍스트를 위해 무언가를 **읽기**만 하면 되는가?" → Resource.

> **CCA 시험 팁:** Tool vs Resource 경계가 초기 응시자들에 따르면 **가장 높은 예상 외 점수 손실**을 유발합니다.

구조화된 데이터 추출 파이프라인에서 신뢰성의 세 단계를 가장 낮은 것부터 높은 순서로 나열하면?

D) few-shot 예시가 도구 설명의 필요성을 대체한다

**정답: B**

**해설:** few-shot 예시는 **원하는 출력 형식과 품질의 시연** 역할을 합니다. 좋은 출력이 어떻게 생겼는지 보여줍니다. 하지만 도구 실행 순서를 **신뢰할 수 있게 제어하지는 못합니다** — 그것은 결정론적 강제를 위해 `tool_choice`나 hooks가 필요합니다.

### 💡 문제 60 (Hard)
아키텍트가 다음을 충족해야 하는 프로덕션 시스템을 설계합니다: (1) 고객 요청을 실시간 처리, (2) 금융 거래의 컴플라이언스 규칙 강제, (3) 비용 최적화, (4) 감사 로그 유지. 올바른 접근의 조합은?

A) Batch API + 프롬프트 기반 컴플라이언스 + 대화 히스토리 로그
B) Real-Time API + 시스템 프롬프트 규칙 + 수동 감사 검토
C) **Prompt Caching이 적용된 Real-Time API + 컴플라이언스를 위한 PreToolUse hooks + 감사 로깅을 위한 PostToolUse callbacks**
D) Prompt Caching이 적용된 Batch API + 스키마 기반 컴플라이언스 + 자동 로그 내보내기

**정답: C**

**해설:** 이 문제는 여러 CCA 개념을 통합합니다: (1) 실시간 처리를 위한 **Real-Time API**(Batch API는 실시간에 절대 적합하지 않음). (2) 반복되는 시스템 프롬프트의 비용 최적화를 위한 **Prompt Caching**(최대 90% 절감). (3) 결정론적 컴플라이언스 강제를 위한 **PreToolUse hooks**(프로그래밍적, 프롬프트 기반이 아닌). (4) 자동적이고 포괄적인 감사 로깅을 위한 **PostToolUse callbacks**. A는 Batch API(실시간에 부적합)와 프롬프트 기반 컴플라이언스(강제에 부적합)를 사용합니다. B는 프로그래밍적 hooks 대신 프롬프트 규칙을 사용합니다. D는 Batch API를 사용합니다.

> **CCA 시험 팁:** 시험 마지막에 나오는 "통합" 문제의 패턴입니다. 5개 도메인을 하나의 일관된 아키텍처로 결합할 수 있는지 테스트합니다.
