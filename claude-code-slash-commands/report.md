# 워크플로우를 바꾸는 Claude Code의 새로운 슬래시 커맨드 5가지

> 원문: [5 New Claude Code Slash Commands (That Are Making Workflows Better)](https://medium.com/@joe.njenga/5-new-claude-code-slash-commands-that-are-making-workflows-better-7bd416a5859a) — Joe Njenga, 2026년 3월 18일경

---

## 1. 요약

Anthropic이 최근 몇 주간 Claude Code에 연이어 추가한 새로운 슬래시 커맨드들을 정리한 글이다. `/effort`(사고 깊이 조절), `/loop`(반복 프롬프트 자동화), `/copy`(코드 블록 선택 복사), `/color`(세션별 색상 구분), 그리고 `/simplify`와 `/batch`(병렬 에이전트 기반 코드 리뷰 및 대규모 마이그레이션)까지 총 6개 이상의 커맨드를 다룬다. 저자는 각 커맨드의 사용법, 활용 시나리오, 그리고 실제 워크플로우에서의 체감 효과를 구체적으로 설명하며, 특히 `/effort`와 `/copy`를 가장 자주 사용한다고 밝힌다. Anthropic의 업데이트 속도가 너무 빨라 놓치기 쉬운 기능들을 한곳에 모아 정리한 실용적인 가이드 성격의 글이다.

## 2. 상세 내용

### 2.1 /effort - Claude의 사고 깊이 조절

이전에는 `/model` 메뉴에 들어가 화살표 키로 슬라이더를 조정해야 했지만, 이제 `/effort`만 입력하면 세 단계를 바로 선택할 수 있다.

| 단계 | 기호 | 설명 |
|------|------|------|
| Low | ○ | 깊은 추론 생략. 빠르고 저렴 |
| Medium | ◐ | 균형 잡힌 모드. 대부분의 코딩 작업에 적합 |
| High | ● | 전체 추론 능력. 복잡한 문제에 최적 |

- `/effort auto`로 기본값 복원 가능 (Claude가 작업 복잡도에 따라 자동 결정)
- Opus 4.6은 Max/Team 구독자 기준 medium이 기본값
- **Claude가 응답 중에도 전환 가능** - 다음 턴부터 적용됨
- 기존의 "ultrathink" 프롬프트 트릭은 1회성이었지만, `/effort`는 세션 전체에 지속 적용

**활용 시나리오:**
- Low: 변수 이름 변경, 코드 포매팅, 간단한 질문
- Medium: 일상적 코딩, 표준 기능 구현
- High: 아키텍처 결정, 복잡한 디버깅, 멀티파일 리팩토링, 코드 리뷰

### 2.2 /loop - 반복 프롬프트 자동화

지정된 간격으로 프롬프트나 슬래시 커맨드를 반복 실행한다.

```
/loop 5m check the deploy     # 5분마다 배포 상태 확인
/loop 10m /stats              # 10분마다 토큰 사용량 추적
```

- 내부적으로 같은 릴리스에 추가된 cron 스케줄링 도구와 연동
- 외부 cron job이나 스크립트 없이 세션 내에서 반복 작업 처리 가능
- 중지하려면 환경변수 `CLAUDE_CODE_DISABLE_CRON=1` 설정

**주의사항:** 가벼운 읽기 전용 체크에 적합. 무거운 코드 생성을 반복하면 토큰을 빠르게 소진한다.

**활용 시나리오:**
- 배포 모니터링
- 테스트 결과 확인
- CI 파이프라인 빌드 실패 감시
- 프로젝트 헬스 체크

### 2.3 /copy - 코드 블록 선택 복사

이전에는 응답에서 코드를 복사하려면 수동으로 스크롤해서 선택하거나 전체 응답을 복사 후 잘라내야 했다. `/copy`는 인터랙티브 선택기를 제공한다.

- 응답 내 모든 코드 블록이 개별 항목으로 나열됨
- 원하는 블록만 골라 클립보드에 복사
- "Always copy full response" 모드를 선택하면 이후 `/copy`가 항상 전체 응답을 복사

**SSH 환경 지원:** 클립보드 접근이 제한된 SSH 환경에서 선택 항목에 `w` 키를 누르면 파일에 직접 저장. tmux나 원격 세션에서의 클립보드 문제를 우회할 수 있다.

### 2.4 /color - 세션별 색상 구분

여러 Claude Code 세션을 동시에 실행할 때 프롬프트 바에 색상을 지정하여 시각적으로 구분한다.

```
/color                    # 색상 선택
/color default            # 초기화 (gray, reset, none도 동일)
```

- `/rename`과 함께 사용하면 효과적 - 세션 이름이 프롬프트 바에 표시되므로, 색상 + 이름("auth-service", "frontend-v2" 등) 조합으로 세션 관리가 편해짐

### 2.5 /simplify - 병렬 에이전트 코드 리뷰

기능 구현 후 사후 정리(post-implementation cleanup) 도구. `git diff`로 변경 사항을 파악한 뒤 **세 개의 전문 에이전트를 병렬로 실행**한다.

| 에이전트 | 역할 |
|----------|------|
| Code reuse agent | 중복 로직 발견 및 통합 |
| Code quality agent | 코드 스멜 감지, CLAUDE.md 규칙 적용 |
| Efficiency agent | 성능 이슈 및 구조 개선 제안 |

- 세 에이전트가 동시에 실행되고, 완료 후 결과를 종합하여 한 번에 수정 적용
- `/simplify focus on error handling`처럼 특정 영역에 집중 가능
- 저자의 테스트: 69줄의 지저분한 코드를 90초 만에 33줄로 축소. raw SQL 문자열 연결 제거, CommonJS를 ES 모듈로 전환, 중첩 루프를 배열 메서드로 교체, 죽은 코드 제거 등을 자동 수행

### 2.6 /batch - 대규모 병렬 마이그레이션

`/simplify`가 코드 품질 검사라면, `/batch`는 마이그레이션 엔진이다.

```
/batch migrate src/ from Jest to Vitest
/batch replace all uses of lodash with native equivalents
/batch add type annotations to all untyped function parameters
```

**3단계 프로세스:**

1. **Research & Plan (조사 및 계획):** 디렉토리 전체를 탐색하고, 변경이 필요한 파일과 이미 준수하는 파일을 구분한 마이그레이션 테이블을 생성. 승인 전까지 코드를 수정하지 않음
2. **Execute in Parallel (병렬 실행):** 작업을 독립 단위로 분해하고, 격리된 git worktree에서 각 에이전트가 개별 코드베이스 복사본을 가지고 작업
3. **Verify & PR (검증 및 PR):** 변경 후 검증 체크를 실행하고, 각 워커가 PR 생성 전 `/simplify`를 실행하여 사전 리뷰된 상태로 풀 리퀘스트를 생성

### 2.7 /reload-plugins - 플러그인 핫 리로드

플러그인 코드를 수정해도 세션을 재시작하지 않으면 반영되지 않던 문제를 해결. `/reload-plugins`를 실행하면 컨텍스트 손실 없이 변경사항이 즉시 적용된다.

## 3. 핵심 인사이트 정리

1. **속도 vs 품질 트레이드오프의 명시적 제어**: `/effort`는 AI의 추론 깊이를 개발자가 직접 조절할 수 있게 하여, 단순 작업에는 비용을 절약하고 복잡한 작업에는 전체 능력을 투입하는 전략적 사용을 가능하게 한다.
2. **세션 내 자동화의 등장**: `/loop`은 외부 도구 없이 Claude Code 세션 안에서 cron-like 반복 작업을 처리할 수 있게 하여, 개발자가 터미널을 전환하지 않고도 모니터링을 유지할 수 있다.
3. **병렬 에이전트가 코드 리뷰의 미래**: `/simplify`와 `/batch`는 단일 AI가 순차적으로 처리하는 방식이 아니라, 역할별 전문 에이전트를 병렬로 투입하는 패턴을 보여준다. 이는 코드 리뷰와 마이그레이션의 속도와 품질을 동시에 높인다.
4. **"Plan-then-Execute" 패턴의 중요성**: `/batch`가 계획을 먼저 보여주고 승인을 기다리는 설계는, 대규모 자동 수정에서 개발자의 통제권을 유지하는 핵심 안전장치다.
5. **DX(개발자 경험) 세부사항의 가치**: `/color`, `/copy`의 SSH 파일 쓰기, `/reload-plugins`의 핫 리로드 같은 소소한 기능이 일상적 워크플로우의 마찰을 크게 줄인다.
6. **Anthropic의 업데이트 속도가 곧 경쟁력**: 기능들이 대형 릴리스가 아닌 패치 노트 속에 조용히 추가되는 방식은, 사용자가 능동적으로 따라가지 않으면 놓치기 쉽다는 것을 의미한다.

## 4. 원문 영어 표현 해설

### 핵심 개념어

| 원문 표현 | 직역 | 저자가 의도한 뉘앙스 |
|-----------|------|---------------------|
| slash commands | 슬래시 커맨드 | CLI에서 `/`로 시작하는 내장 명령어. "커맨드"보다 더 가볍고 즉석에서 쓰는 도구라는 느낌 |
| effort level | 노력 수준 | AI가 문제에 투입하는 연산/추론의 깊이. 인간의 "집중도"에 비유한 직관적 명명 |
| persistent control | 지속적 제어 | 1회성이 아닌 세션 전체에 걸쳐 유지되는 설정. "ultrathink"(1회)과의 대비를 강조 |
| interactive picker | 인터랙티브 선택기 | 목록에서 하나를 골라 선택하는 UI 패턴. 단순 복사가 아닌 "고르는 경험"을 부각 |
| post-implementation cleanup | 구현 후 정리 | 기능이 동작하는 상태에서 코드 품질을 개선하는 단계. "리팩토링"보다 가벼운 뉘앙스 |
| migration engine | 마이그레이션 엔진 | 대규모 코드 변환을 자동화하는 도구. "engine"이라는 단어로 강력함과 체계성을 암시 |
| decompose | 분해하다 | 큰 작업을 독립적인 작은 단위로 나누는 것. 소프트웨어 엔지니어링의 핵심 설계 원칙을 환기 |

### 비유/수사 표현

| 원문 표현 | 문맥 | 뉘앙스 |
|-----------|------|--------|
| "shipping at a pace that's hard to match" | Anthropic의 업데이트 속도 묘사 | "ship"은 소프트웨어 출시의 업계 은어. "배를 띄우다"에서 온 표현으로, 제품을 세상에 내보내는 행위. "hard to match"는 경쟁사도 사용자도 따라가기 힘들다는 이중적 의미 |
| "quietly made in patch notes that move fast and pile up" | 기능이 조용히 추가되는 상황 | "pile up"(쌓이다)은 빨래나 서류가 쌓이듯 처리되지 않은 정보가 누적되는 이미지. 한국어의 "밀린 숙제" 같은 느낌 |
| "buried inside a menu" | 이전 effort 설정 방식 | "묻히다" - 찾기 어려운 곳에 숨겨져 있다는 뜻. UX 비판에 자주 쓰이는 표현 |
| "burn through your tokens" | 토큰 과다 소비 경고 | "태워버리다" - 돈이나 자원을 빠르게 소진하는 것. "burn through cash"(돈을 펑펑 쓰다)에서 온 관용구 |
| "a long afternoon of find-and-replace" | /batch 없이 수동 마이그레이션 | 지루하고 반복적인 오후 작업. 개발자라면 누구나 겪어본 고통을 환기시키는 공감 유도 표현 |
| "hoping you didn't miss anything" | 수동 작업의 불안감 | 수작업 마이그레이션의 본질적 불안감. /batch의 체계적 접근과 대비시키는 수사 장치 |
| "spawns agents in isolated git worktrees" | /batch의 병렬 실행 설명 | "spawn"은 게임/프로그래밍에서 새 엔티티를 생성하는 행위. 에이전트가 각자의 격리된 환경에서 태어나 독립적으로 일한다는 이미지 |

### 업계 전문 용어

| 원문 표현 | 의미 | 왜 이 단어를 쓰는가 |
|-----------|------|---------------------|
| code smells | 코드 악취 | Martin Fowler가 대중화한 용어. 당장 버그는 아니지만 설계상 문제가 있음을 감각적으로 표현. "냄새가 난다 = 뭔가 잘못됐다" |
| git worktree | Git 작업 트리 | 하나의 리포지토리에서 여러 브랜치를 동시에 체크아웃할 수 있는 Git 기능. /batch가 병렬 작업의 격리를 위해 활용 |
| dead code | 죽은 코드 | 실행되지 않는 코드. "살아있는" 코드와 대비하여 제거 대상임을 직관적으로 전달 |
| hot-reload | 핫 리로드 | 프로세스를 재시작하지 않고 변경사항을 즉시 반영. 프론트엔드 개발(HMR)에서 익숙한 개념을 플러그인에 적용 |
| token usage | 토큰 사용량 | LLM API 과금의 기본 단위. AI 개발 시대의 새로운 "비용 지표"로, 전통적 서버 비용 개념을 대체 |
| CI pipeline | 지속적 통합 파이프라인 | 코드 변경 시 자동으로 빌드/테스트를 실행하는 자동화 체계. DevOps의 핵심 인프라 |
| DX (Developer Experience) | 개발자 경험 | UX(사용자 경험)의 개발자 버전. 도구가 개발자에게 주는 사용 편의성과 만족감 |

### 저자의 어투에서 느껴지는 것

Joe Njenga의 글쓰기는 **"경험 많은 동료가 커피를 마시며 알려주는" 톤**이다. 기술 문서가 아닌 대화체로, "I understand", "you'll appreciate it", "Let me walk you through"처럼 독자와 직접 대화하는 2인칭 표현을 자주 사용한다.

구조적으로는 **"문제 → 해결 → 활용"** 패턴을 반복한다. 모든 커맨드를 소개할 때 "이전에는 이랬는데(불편), 이제는 이렇다(해결)"로 시작하여, 독자가 즉시 가치를 체감하도록 유도한다. 한국어로 치면 "~했잖아요, 근데 이제는~" 식의 구어체 설명법이다.

수사적으로 **구체적 숫자를 즐겨 사용**한다. "69줄을 33줄로", "90초 만에", "26,000+ 개발자" 등은 주장에 신뢰를 더하는 장치다. 또한 "Quick Note", "Use Cases" 같은 레이블로 정보를 시각적으로 분리하여, 스캔 읽기(skimming)에 최적화된 구조를 만든다.

글 말미에 자신의 유료 강의 홍보를 자연스럽게 배치하는 것은 Medium 테크 블로거들의 전형적 패턴이며, "making workflows better"라는 제목 자체가 검색 최적화(SEO)를 의식한 실용 키워드 조합이다.
