# Anthropic 하네스 엔지니어링 — 오디오 스크립트

오늘은 Anthropic의 하네스 엔지니어링에 대해 이야기해 볼게요. 두 개의 에이전트가 하나의 피처 리스트를 처리하면서 컨텍스트 오버플로를 어떻게 방지하는지, 그 구조가 정말 흥미로워요.

Rick Hightower라는 개발자가 Anthropic의 공식 엔지니어링 문서를 분석한 내용인데요, 핵심 주제는 "장시간 AI 코딩 작업에서 왜 모델이 맥락을 잃어버리는가"예요. 같은 모델이 200줄짜리 함수는 완벽하게 작성하면서, 2천 줄짜리 피처를 구현하다가는 중간에 방향을 잃어버리는 현상이 있거든요.

이 문제를 "컨텍스트 윈도우 경계"라고 불러요. 토큰 한계 자체보다 더 중요한 건, 그 한계가 가까워질 때 모델의 행동이 바뀐다는 점이에요. Claude Sonnet 4.5에서 특히 두드러지는 현상인데, 토큰이 거의 다 채워지면 모델이 작업을 서둘러 마무리하려는 성급한 행동을 보여요. 저자는 이걸 "컨텍스트 불안"이라고 표현했어요. 인간의 마감 불안과 비슷한 거죠. 그리고 단순히 요약하거나 압축하는 방식으로는 이 문제를 해결할 수 없고, 컨텍스트를 완전히 리셋하는 게 필요해요.

Anthropic이 제시한 해법이 바로 2-에이전트 하네스 아키텍처예요. Justin Young을 비롯한 Anthropic 엔지니어링 팀이 공식 문서로 발표한 패턴인데요, 두 가지 전용 에이전트를 조합해요.

첫 번째는 초기화 에이전트예요. 프로젝트의 첫 번째 세션에서만 실행되고, 이후 모든 세션을 위한 인프라를 구축해요. 이 에이전트가 만드는 산출물이 네 가지인데요. 200개 이상의 세분화된 피처 목록을 담은 JSON 파일, Git 초기 커밋, 에이전트 작업 로그 파일, 그리고 개발 서버를 시작하는 초기화 스크립트예요. 처음에 피처 목록의 모든 항목은 "미완료" 상태로 설정돼요.

두 번째는 코딩 에이전트예요. 이후 세션마다 실행되고, 한 번에 하나의 피처를 구현해요. 세션이 시작될 때마다 정해진 프로토콜이 있어요. 작업 디렉토리를 확인하고, Git 로그와 진행 상태 파일을 읽어서 이전 세션에서 무엇을 했는지 파악하고, 개발 서버를 기동한 다음, 아직 완료되지 않은 피처 중 가장 우선순위가 높은 것을 선택해서 작업을 시작해요. 세션이 끝날 때는 완료한 피처를 JSON에서 완료 상태로 바꾸고, 설명적인 메시지와 함께 Git 커밋을 남기고, 진행 상태 파일을 업데이트해요.

이 구조에서 JSON 피처 리스트의 역할이 특히 흥미로워요. 단순한 체크리스트가 아니거든요. 세 가지 핵심 기능을 해요.

첫째, 조기 완료를 방지해요. 200개 이상의 피처가 미완료 상태로 남아있는 한, 에이전트는 "다 했다"고 선언할 수 없어요. 둘째, 새로운 세션이 시작될 때마다 피처 리스트를 읽으면서 전체 요구사항을 다시 파악하는 컨텍스트 앵커 역할을 해요. 닻처럼 에이전트가 표류하지 않도록 붙잡아두는 거죠. 셋째, 무결성을 보호해요. 피처를 삭제하거나 임의로 편집하는 게 금지되어 있어서, 빠뜨리거나 버그 있는 기능을 숨기는 행위를 원천 차단해요.

저자는 이 아키텍처가 GAN, 즉 생성적 적대 신경망과 유사한 구조를 가진다고 분석해요. GAN이 생성자와 판별자를 분리하듯이, 코드를 만드는 역할과 평가하는 역할을 분리하는 거예요. 핵심 통찰은 이거예요. "자신이 만든 작업물을 평가하도록 요청하면, AI 모델은 병적인 낙관주의자가 된다." 생성과 평가를 분리하면 더 의미 있는 피드백 루프가 만들어져요.

테스트 전략도 달라졌어요. 기존의 단위 테스트 중심에서, 브라우저를 직접 조작하는 엔드-투-엔드 검증으로 전환했어요. Playwright 같은 브라우저 자동화 도구를 사용하면 에이전트가 사람처럼 화면을 보면서 검증하기 때문에, 단위 테스트가 놓치는 버그를 잡을 수 있거든요.

실제 비교 사례를 보면 이 접근법의 효과가 더 명확해져요. 단일 에이전트로 20분, 9달러를 써서 작업했을 때는 게임 메카닉이 작동하지 않는 결과물이 나왔어요. 반면 2-에이전트 하네스로 6시간, 200달러를 쓴 경우에는 16개 피처가 모두 작동하고 물리 엔진까지 제대로 구현된 완성도 높은 결과물이 나왔어요. 22배 비용 차이지만, 산출물의 품질 격차는 그 이상이에요. 재작업과 디버깅, 요구사항 재수집에 드는 숨겨진 비용까지 생각하면, 오히려 하네스 방식이 더 경제적인 선택이에요.

정리해 볼게요. 이 아키텍처가 보여주는 핵심은 여러 가지예요.

컨텍스트를 완전히 리셋하는 게 단순 압축보다 효과적이에요. AI 모델의 자기 평가는 구조적으로 신뢰할 수 없기 때문에 생성과 평가를 분리해야 해요. 피처 리스트는 세션 간 기억을 대체하는 외부 장치예요. 그리고 Git은 단순한 버전 관리가 아니라 에이전트의 외부 기억이에요. 각 피처마다 설명적인 커밋 메시지를 남기면, 새 세션이 이전 에이전트의 결정 맥락을 즉시 파악할 수 있거든요.

Anthropic의 하네스 엔지니어링은 단순한 멀티 에이전트 시스템이 아니에요. 컨텍스트 관리라는 근본적인 문제를 구조적으로 해결한 설계예요. 그리고 모델 능력이 개선될수록, 어떤 하네스 요소가 여전히 필요하고 어떤 요소가 불필요한 오버헤드인지 계속 재평가해야 한다는 점도 중요한 통찰이에요. 오늘 이야기는 여기까지예요. 들어주셔서 감사합니다.
