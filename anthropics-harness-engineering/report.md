## 1. 요약

## 2. 상세 내용

### 2.1 문제 정의: 컨텍스트 윈도우 경계

저자는 아젠틱 코딩에서 가장 어려운 문제로 "컨텍스트 윈도우 경계"를 꼽는다.

- 짧은 작업: 모델이 완전한 컨텍스트를 유지한 채 처음부터 끝까지 실행
- 긴 작업: 컨텍스트가 채워질수록 모델의 집중력이 분산되고, 이전 결정을 망각하거나 모순된 선택을 하게 됨
- **컨텍스트 불안(Context Anxiety)**: Claude Sonnet 4.5에서 특히 두드러진 현상. 토큰 한계가 가까워지면 모델이 작업을 서둘러 마무리하려는 성급한 행동을 보임
- 단순 컴팩션(compaction)만으로는 충분하지 않고, 컨텍스트 완전 리셋(reset)이 필요함

### 2.2 Anthropic의 해법: 2-에이전트 하네스 아키텍처

Anthropic 엔지니어링 팀(Justin Young 등)이 공식 문서로 발표한 패턴으로, 두 가지 전용 에이전트를 조합한다.

#### 초기화 에이전트 (Initializer Agent)

첫 번째 세션에서만 실행되며 이후 모든 세션을 위한 인프라를 구축한다.

| 생성 산출물 | 역할 |
|------------|------|
| `feature_list.json` | 200개 이상의 세분화된 피처 목록. 초기 상태: `"passes": false` |
| `git` 초기 커밋 | 초기 파일 구조를 버전 관리에 등록 |
| `claude-progress.txt` | 에이전트 작업 로그. 다음 세션의 시작점 역할 |
| `init.sh` | 개발 서버 시작 등 세션 초기화 스크립트 |

#### 코딩 에이전트 (Coding Agent)

이후 세션마다 실행되며 한 번에 하나의 피처를 구현한다.

**세션 시작 프로토콜:**
1. `pwd` 실행 — 작업 디렉토리 확인
2. Git 로그 및 `claude-progress.txt` 읽기 — 이전 작업 내용 파악
3. `init.sh` 실행 — 개발 서버 기동
4. 기본 동작 검증
5. 완료되지 않은 피처 중 최우선 항목 선택

**세션 종료 규칙:**
- 완료된 피처를 `feature_list.json`에서 `passes: true`로 업데이트
- 설명적인 메시지로 Git 커밋
- `claude-progress.txt` 업데이트

### 2.3 JSON 피처 리스트의 역할

단순한 체크리스트가 아니다. 세 가지 핵심 기능을 수행한다.

1. **조기 완료 방지**: 200개 이상의 피처가 `passes: false`로 남아있는 한, 에이전트는 "완료"를 선언할 수 없음
2. **컨텍스트 앵커**: 새로운 세션이 시작될 때마다 피처 리스트를 읽음으로써 전체 요구사항을 재파악
3. **무결성 보호**: 피처 삭제나 임의 편집 금지 — 빠뜨리거나 버그 있는 기능을 숨기는 행위를 원천 차단

### 2.4 GAN 유사 구조: 생성자-평가자 분리

Hightower는 Anthropic의 아키텍처가 GAN(Generative Adversarial Network)과 유사한 대립 구조를 가진다고 분석한다.

- **핵심 통찰**: "자신이 만든 작업물을 평가하도록 요청하면, AI 모델은 병적인 낙관주의자가 된다"
- 생성과 평가를 분리하면 더 의미있는 피드백 루프가 형성됨
- UI/디자인 태스크: Playwright MCP로 실제 웹 페이지를 조작하며 4가지 기준으로 평가 (디자인 품질, 독창성, 공예성, 기능성)
- 풀스택 개발: Planner → Generator → Evaluator의 3-에이전트 구조로 확장

### 2.5 테스트 전략의 변화

브라우저 자동화 도구(Puppeteer MCP) 도입이 성과에 결정적 영향을 미쳤다. 단위 테스트 중심에서 사람처럼 화면을 보고 조작하는 엔드-투-엔드 검증으로 전환했다.

### 2.6 비용 대비 품질: $9 vs $200

| 방식 | 소요 시간 | 비용 | 결과 |
|------|----------|------|------|
| 단일 에이전트 | 20분 | $9 | 게임 메카닉 비작동, 사용 불가 |
| 2-에이전트 하네스 | 6시간 | $200 | 16개 피처, 물리 엔진 작동, 플레이 가능 |

22배의 비용 차이지만 산출물의 품질 격차는 그 이상이다. 저자는 이를 "하네스 설계가 단순한 구현 세부사항이 아니라 전략적 결정"임을 보여주는 근거로 활용한다.

## 3. 핵심 인사이트 정리

1. **컨텍스트 리셋이 컴팩션보다 효과적**: Claude Sonnet 4.5의 컨텍스트 불안은 컴팩션으로 완화되지 않았다. 컨텍스트를 완전히 초기화하고 정형화된 상태 파일에서 다시 시작하는 것이 더 우수한 결과를 냈다.

2. **자기 평가는 신뢰할 수 없다**: AI 모델은 자신의 작업물을 평가할 때 구조적으로 관대해진다. 생성 역할과 평가 역할을 별도 에이전트로 분리하는 것이 품질 루프의 핵심이다.

3. **피처 리스트는 기억 보조 장치다**: 세션 간 무엇을 해야 하는지를 외부 파일로 관리하면, 모델은 컨텍스트에 의존하지 않고도 전체 목표를 유지할 수 있다.

4. **Git은 단순 버전 관리가 아닌 에이전트의 외부 기억이다**: 각 피처마다 설명적 커밋 메시지를 남기면, 새 세션이 `git log`로 이전 에이전트의 결정 맥락을 즉시 파악할 수 있다.

5. **하네스의 모든 컴포넌트는 모델이 혼자 못 하는 것에 대한 가정을 담는다**: 모델 능력이 개선될수록 어떤 하네스 요소가 여전히 필요하고 어떤 요소가 불필요한 오버헤드인지 재평가해야 한다.

6. **브라우저 자동화 도구는 테스트의 질을 근본적으로 바꾼다**: Puppeteer MCP처럼 실제 브라우저를 조작하는 도구를 에이전트에게 주면, 사람처럼 화면을 보고 검증하므로 단위 테스트가 놓치는 버그를 잡는다.

7. **$200짜리 하네스가 $9짜리 단일 에이전트보다 저렴하다**: 재작업, 디버깅, 요구사항 재수집에 드는 비용을 계산하면 품질 보장 구조에 투자하는 것이 장기적으로 경제적이다.

### 저자 어투 분석

Rick Hightower는 **실전 경험 기반의 전도사 스타일**로 글을 쓴다. 이론을 나열하기보다 "왜 기존 방법이 실패하는가"에서 시작해 구체적 사례($9 vs $200)로 독자를 설득한 후 솔루션을 제시하는 3단 구조를 반복한다.

- **수사 패턴**: 수사 의문문으로 문제를 제기("Why does the same model...?") → 독자의 경험과 공명시킨 후 → 해답 제시

- **시리즈 전략**: 각 글이 독립적으로 읽히면서도 "Harness Engineering" 시리즈 내에서 누적 학습이 이뤄지도록 개념을 점층적으로 도입
