# CCA CI/CD 시나리오 오디오 스크립트

> 예상 분량: 약 12분 (약 3,800자)

---

CCA 딥다이브 시리즈 다섯 번째 에피소드입니다. 오늘은 가장 실무적인 CI/CD 시나리오입니다. 정확한 플래그와 구문을 외워야 하는, 시리즈에서 유일한 암기형 시나리오이기도 합니다.

저자 Rick Hightower도 직접 "Memorize the details"라고 말합니다. 다른 시나리오에서는 "이해하라"고 했는데, 이 시나리오만큼은 "외워라"입니다. 그만큼 구체적인 플래그와 구문이 시험에 직접 나온다는 뜻이죠.

---

자, 그럼 CI/CD 파이프라인에서 Claude Code를 돌릴 때 **반드시** 알아야 할 세 가지 플래그부터 시작하겠습니다.

첫 번째, 대시 p 플래그입니다. 대시 p는 대시대시 print의 줄임말이에요. 이게 없으면 어떻게 될까요? 파이프라인이 에러를 내는 것도 아니고, 느려지는 것도 아닙니다. 그냥 멈춰요. 영원히 입력을 기다리는 상태가 됩니다. 원문에서 이걸 아주 극적으로 표현하는데요, "Not slowly, not with an error. It just sits there, waiting for input that will never arrive." 절대 오지 않을 입력을 기다리면서 그냥 거기 앉아 있다는 겁니다. 세 문장으로 점점 절망적인 상황을 그리는 점강법이 인상적이죠.

여기서 시험 함정이 나옵니다. "파이프라인 타임아웃을 120분으로 늘려라"라는 선택지가 나오면? **오답**입니다. 그건 증상만 늦추는 거지, 원인을 해결하는 게 아니에요. 행의 원인은 대시 p 누락이니까요.

두 번째, 대시대시 bare 플래그입니다. 이건 뭘 하느냐면, 훅, LSP, 플러그인 동기화, 스킬 디렉토리 스캔, 자동 메모리, OAuth 인증 같은 것들을 전부 건너뜁니다. 최소한의 헤드리스 모드라고 보시면 됩니다.

왜 필요할까요? 대시 p만 쓰고 대시대시 bare를 안 쓰면 어떻게 되느냐. 개발자 로컬 환경에서는 개인 CLAUDE.md 파일, MCP 서버, 훅 같은 것들이 로드됩니다. 그런데 CI 러너에서는 이런 게 없잖아요. 결과적으로 **같은 명령어인데 다른 동작**이 나옵니다. 재현성이 깨지는 거예요.

Anthropic 공식 문서에서도 대시대시 bare를 스크립트와 SDK 호출의 권장 모드라고 명시하고 있고, 앞으로 대시 p의 기본값이 될 예정이라고까지 밝혔습니다. 그러니까 이 두 개는 세트입니다. 대시 p 더하기 대시대시 bare, 이게 CI/CD의 기본 조합이에요. 하나만 쓰면 안 됩니다.

한 가지 주의할 점. 대시대시 bare는 OAuth와 키체인 인증을 건너뛰기 때문에, ANTHROPIC_API_KEY 환경변수를 명시적으로 설정해줘야 합니다.

세 번째, 대시대시 output-format json입니다. 기계가 파싱할 수 있는 출력을 보장하는 플래그예요.

여기서 두 가지 안티패턴을 짚고 넘어가겠습니다. 첫째, 자연어 출력을 정규식으로 파싱하는 것. Claude의 출력은 실행할 때마다 포맷이 달라질 수 있어서 간헐적으로 실패합니다. 둘째, 프롬프트에 "항상 JSON으로 응답해"라고 쓰는 것. 이건 90퍼센트는 작동합니다. 하지만 Claude가 마크다운 코드 블록으로 감싸거나 설명 텍스트를 덧붙일 수가 있어요. 원문에서 이걸 명쾌하게 정리합니다. **프롬프트는 안내이고, 플래그는 보장이다.** 시험은 이 차이를 명확히 테스트합니다.

---

여기에 대시대시 json-schema까지 결합하면 출력 구조를 완전히 제약할 수 있습니다. 예를 들어 issues 배열과 approve 불리언을 가진 JSON 객체를 스키마로 지정할 수 있어요. 이때 중요한 포인트, 결과값은 result 필드가 아니라 **structured_output 필드**에 담깁니다. 시험에서 이 필드명을 물어볼 수 있으니 기억해두세요.

---

다음으로 도구 샌드박싱입니다. 두 가지 플래그가 있는데, 이름이 비슷해서 헷갈리기 쉽습니다.

대시대시 tools는 사용 가능한 도구를 **제한**합니다. 예를 들어 대시대시 tools에 Read 콤마 Bash만 지정하면, Claude는 그 두 도구만 쓸 수 있어요. 이게 진짜 샌드박싱이죠.

반면에 대시대시 allowedTools는 해당 도구를 권한 프롬프트 없이 **사전 승인**하는 겁니다. 도구를 제한하는 게 아니에요. CI/CD에서 의도하지 않은 동작을 방지하려면? 대시대시 tools를 써야 합니다. 시험에서 "의도하지 않은 동작 방지"라는 표현이 나오면 답은 대시대시 tools입니다.

---

이제 파이프라인 아키텍처 패턴으로 넘어갑니다. 시험에서 가장 자주 나오는 건 자동 PR 코드 리뷰 패턴이에요. GitHub Actions에서 PR diff를 가져와서, Claude에게 보안 이슈를 리뷰하게 하는 거죠. 이때 대시대시 bare, 대시 p, 대시대시 output-format json, 대시대시 tools를 전부 조합해서 씁니다.

두 번째 패턴은 자동 테스트 생성인데, 이건 보통 야간에 돌리니까 Batch API 후보가 됩니다.

세 번째는 수정 파이프라인이에요. 테스트가 실패하면 Claude로 수정 시도하고, 테스트를 재실행하고, 또 실패하면 에러 피드백과 함께 재시도합니다. 여기서 핵심은 재시도 횟수를 **최대 2에서 3회로 제한**한다는 겁니다. 무한 재시도는 안티패턴이에요. 그리고 구체적인 에러 피드백 없이 맹목적으로 재시도하는 것도 안티패턴입니다.

---

자, 이제 토큰 경제학입니다. 이 부분은 숫자가 나오니까 정확히 외워야 합니다.

Prompt Caching의 비용 구조를 보겠습니다. 캐시 읽기는 기본 비용의 0.1배, 즉 **90퍼센트 절감**입니다. 이게 핵심 절감 포인트예요. 캐시 쓰기는 5분 TTL 기준 기본의 1.25배, 25퍼센트가 추가됩니다. 1시간 TTL로 가면 기본의 2배, 100퍼센트 추가예요.

여기서 시험 함정이 또 나옵니다. "Prompt Caching으로 모든 토큰이 90퍼센트 절감된다"라는 선택지. **오답**이에요. 90퍼센트 절감은 캐시 읽기 토큰에만 적용됩니다. 쓰기 토큰은 오히려 비용이 더 듭니다.

---

그리고 CCA 시험에서 정말 중요한 사실 하나. **Message Batches API는 Zero Data Retention, 줄여서 ZDR 대상이 아닙니다.**

ZDR은 뭐냐면, 데이터가 처리된 후 보유되지 않는 정책이에요. 의료, 금융, 정부처럼 규제가 엄격한 산업에서 필수로 요구됩니다. 그런데 Batch API는 이 ZDR을 지원하지 않아요. 그래서 규제 산업 고객은 야간 분석이라 해도 Batch API를 쓸 수 없고, Real-Time API에 Prompt Caching을 조합해서 써야 합니다.

시험에서 규제 산업의 비용 절감 문제가 나왔는데 선택지에 "Batch API"가 있다? 즉시 제거하세요.

의사결정 프레임워크로 정리하면 이렇습니다. PR 리뷰나 배포 게이트처럼 개발자가 대기하는 상황은 Real-Time API에 Prompt Caching, ZDR 호환 됩니다. 야간 테스트 생성인데 ZDR이 필요 없으면 Batch API에 Caching을 조합할 수 있어요. 하지만 야간 분석이라도 ZDR이 필요하면 Real-Time API로 가야 합니다.

---

마지막으로 안티패턴을 총정리하겠습니다. 아홉 가지인데, 시험에서 이 중 하나가 선택지로 나올 확률이 높으니 빠르게 훑어볼게요.

대시 p 누락은 파이프라인 행. 대시대시 bare 누락은 CI와 로컬의 동작 차이. 정규식 파싱은 간헐적 실패. 프롬프트 전용 JSON은 비JSON 텍스트 혼입. 블로킹 워크플로우에 Batch API를 쓰면 개발자가 시간 단위로 대기. ZDR 환경에서 Batch API는 컴플라이언스 위반. 유효성 검사 없이 Claude 출력을 맹신하는 것. 재시도 없이 첫 에러에 파이프라인을 실패시키는 것. 대시대시 tools 누락으로 Claude가 의도하지 않은 동작을 하는 것.

---

정리하겠습니다. CI/CD 시나리오의 핵심은 딱 세 줄입니다.

첫째, 대시 p 더하기 대시대시 bare는 세트다. 둘 다 써야 비인터랙티브이면서 재현 가능하다.

둘째, 프롬프트로 JSON을 요청하는 것과 플래그로 강제하는 것은 다르다. 시험은 이 차이를 반드시 묻는다.

셋째, Batch API는 ZDR 대상이 아니다. 규제 산업 문제에서 Batch API가 보이면 제거하라.

이 세 가지만 확실히 기억하면, CI/CD 시나리오는 고득점을 노릴 수 있습니다.

다음 에피소드에서는 구조화된 데이터 추출 시나리오를 다루겠습니다.
